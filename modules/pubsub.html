Publish/Subscribe

Send and receive messages reliably with matching QoS policies.

CPU-Light Required

What / Why / When

What

The publish/subscribe pattern decouples data producers from consumers via named topics. Publishers send messages without knowing who receives them; subscribers receive all messages on topics they join. This is ROS2's primary communication mechanism for sensor data, commands, and state updates.

Why

Pub/sub enables flexible system composition where nodes can be added or removed without modifying existing code. It scales to many-to-many connections and supports different QoS profiles for reliability vs. latency trade-offs. This pattern is essential for streaming data like laser scans, camera images, and odometry.

When

Use pub/sub for continuous data streams, sensor readings, and state broadcasts. Avoid it for request/response patterns (use services instead) or long-running tasks requiring feedback (use actions). If message order or delivery guarantees matter, configure appropriate QoS settings.

10-Minute Drill

C++ Micro-Task

Create a minimal string publisher that sends "Hello ROS2" at 1 Hz:

#include "rclcpp/rclcpp.hpp"
#include "std_msgs/msg/string.hpp"

class MinimalPublisher : public rclcpp::Node {
public:
  MinimalPublisher() : Node("minimal_publisher") {
    publisher_ = this->create_publisher<std_msgs::msg::String>(____, 10);
    timer_ = this->create_wall_timer(
      std::chrono::seconds(1),
      std::bind(&MinimalPublisher::timer_callback, this));
  }

private:
  void timer_callback() {
    auto message = std_msgs::msg::String();
    message.data = "____";
    RCLCPP_INFO(this->get_logger(), "Publishing: '%s'", message.data.c_str());
    publisher_->____(____(message));
  }
  
  rclcpp::Publisher<std_msgs::msg::String>::SharedPtr publisher_;
  rclcpp::TimerBase::SharedPtr timer_;
};

int main(int argc, char *argv[]) {
  rclcpp::init(argc, argv);
  rclcpp::spin(std::make_shared<MinimalPublisher>());
  rclcpp::shutdown();
  return 0;
}

Python Micro-Task

Create a matching subscriber that echoes received messages:

import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class MinimalSubscriber(Node):
    def __init__(self):
        super().__init__('minimal_subscriber')
        self.subscription = self.create_subscription(
            String,
            '____',
            self.listener_callback,
            ____)
        
    def listener_callback(self, msg):
        self.get_logger().info('I heard: "%s"' % msg.____)

def main(args=None):
    rclpy.init(args=args)
    minimal_subscriber = MinimalSubscriber()
    rclpy.____(minimal_subscriber)
    minimal_subscriber.destroy_node()
    rclpy.shutdown()

if __name__ == '__main__':
    main()

Concept ↔ Code (Python vs C++)

Publisher Pattern

C++ Python 
auto pub = this->create_publisher<
  std_msgs::msg::String>(
  "chatter", 10);
 
self.publisher = self.create_publisher(
    String, 'chatter', 10)

Subscriber Pattern

C++ Python 
auto sub = this->create_subscription<
  std_msgs::msg::String>(
  "chatter", 10,
  std::bind(&Node::callback, this, _1));
 
self.subscription = self.create_subscription(
    String, 'chatter', self.callback, 10)

Publishing Messages

C++ Python 
auto msg = std_msgs::msg::String();
msg.data = "hello";
publisher_->publish(msg);
 
msg = String()
msg.data = 'hello'
self.publisher.publish(msg)

Parameters & Config

QoS Profile (YAML)

chatter_publisher:
  ros__parameters:
    qos_overrides:
      /chatter:
        publisher:
          # Keep last 10 messages in queue
          depth: 10
          # Best-effort (0) or Reliable (1)
          reliability: 1
          # Volatile (0) or Transient-Local (1) - keep last message for late joiners
          durability: 0
          # Deadline in nanoseconds (0 = infinite)
          deadline: 0

Common Failures & Fix Patterns

Failure 1: No Messages Received

Symptom: Subscriber runs but ros2 topic echo shows nothing.

Root Cause: QoS mismatch between publisher and subscriber. Publisher uses reliable, subscriber uses best-effort.

Fix: Match QoS profiles or use SystemDefaultQoS for both:

// C++ - Use explicit reliable QoS
auto qos = rclcpp::QoS(rclcpp::KeepLast(10)).reliable();
publisher_ = this->create_publisher<std_msgs::msg::String>("chatter", qos);
subscription_ = this->create_subscription<std_msgs::msg::String>("chatter", qos, callback);

Failure 2: Late-Joining Subscribers Miss Data

Symptom: Subscriber started after publisher misses initial messages.

Root Cause: Volatile durability policy discards messages for late joiners.

Fix: Use transient-local durability to keep last N messages:

# Python - Keep last message for late subscribers
from rclpy.qos import QoSProfile, DurabilityPolicy
qos = QoSProfile(depth=10, durability=DurabilityPolicy.TRANSIENT_LOCAL)
self.publisher = self.create_publisher(String, 'state', qos)

Failure 3: Message Queue Overflow

Symptom: Warning "queue depth exceeded" and dropped messages.

Root Cause: Publisher sends faster than subscriber processes, queue size too small.

Fix: Increase queue depth or throttle publisher rate:

// Increase queue depth from 10 to 100
auto pub = this->create_publisher<sensor_msgs::msg::LaserScan>("scan", 100);

Failure Injection Drill

Exercise: QoS Mismatch Detection

Intentionally create a QoS incompatibility and observe the failure:

# Publisher with RELIABLE
from rclpy.qos import QoSProfile, ReliabilityPolicy
qos_pub = QoSProfile(depth=10, reliability=ReliabilityPolicy.RELIABLE)
self.publisher = self.create_publisher(String, 'test', qos_pub)

# Subscriber with BEST_EFFORT
qos_sub = QoSProfile(depth=10, reliability=ReliabilityPolicy.BEST_EFFORT)
self.subscription = self.create_subscription(String, 'test', self.callback, qos_sub)

Detection: Run both nodes, then check:

ros2 topic info /test --verbose
# Look for "Incompatible QoS Policies" warning

Recovery: Change subscriber to RELIABLE or publisher to BEST_EFFORT to match.

Warning:

QoS mismatches don't always throw errors. Nodes may run silently without communicating. Always verify with ros2 topic info --verbose.
