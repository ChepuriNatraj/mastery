Recovery Behaviors

Unstick robots with systematic recovery actions when navigation fails.

Nav2 Fault Handling

What / Why / When

What

Recovery behaviors are atomic actions executed when navigation fails. Nav2 provides built-in recoveries: Spin (rotate in place), BackUp (reverse), Wait (pause), ClearCostmap (reset obstacle data). Each recovery is a ROS2 action server. Behavior trees trigger recoveries through RecoveryNode with configurable retry counts. Recovery sequences chain multiple actions (e.g., clear costmap → spin → retry navigation). Recovery plugins are modular - add custom behaviors like LED signals, sound alerts, or request human help. Recoveries prevent complete mission failure by resolving local stuck conditions.

Why

Navigation fails for recoverable reasons: temporary sensor glitches (clear costmap), robot orientation issues (spin), insufficient clearance (backup). Without recoveries, single failures abort missions. Recovery behaviors provide systematic debugging - each action tests a hypothesis about failure cause. Spin helps with localization uncertainty. BackUp creates space in tight spots. Wait allows dynamic obstacles to clear. Clearing costmaps removes phantom obstacles from sensor noise. Recovery data (which actions worked) improves mission planning.

When

Use recoveries in all navigation BT trees - wrap FollowPath with RecoveryNode. Configure retry counts based on environment: tight spaces need fewer retries (3) to avoid oscillation, open areas can try more (5-6). Add timeout-based recoveries for stuck detection. Use clearing behaviors after sensor errors or kidnapped robot. Don't over-rely on recoveries - if spinning 6 times doesn't work, the issue is deeper (bad map, hardware failure). Log recovery triggers to identify navigation weakness patterns.

Built-in Recovery Actions

Recovery Purpose Parameters Use Case
Spin Rotate in place spin_dist (radians) Localization uncertainty, sensor refresh
BackUp Reverse direction backup_dist (m), backup_speed Stuck in narrow passage, too close to obstacle
Wait Pause execution wait_duration (sec) Dynamic obstacles, temporary blockage
ClearCostmap Reset obstacle data service_name Sensor glitches, phantom obstacles
DriveOnHeading Move straight distance dist_to_travel, speed Escape local minimum
AssistedTeleop Human joystick control time_allowance Complex stuck situations

Calling Recovery Action (Python)

#!/usr/bin/env python3
import rclpy
from rclpy.action import ActionClient
from rclpy.node import Node
from nav2_msgs.action import Spin
import math

class RecoveryClient(Node):
    def __init__(self):
        super().__init__('recovery_client')
        self.spin_client = ActionClient(self, Spin, '____')  # /spin
    
    def send_spin_goal(self, angle_radians):
        # Wait for action server
        self.spin_client.wait_for_server()
        
        # Create goal
        goal = Spin.Goal()
        goal.target_yaw = angle_radians
        
        # Send goal
        future = self.spin_client.send_goal_async(
            goal,
            feedback_callback=self.feedback_callback
        )
        future.add_done_callback(self.goal_response_callback)
    
    def feedback_callback(self, feedback_msg):
        feedback = feedback_msg.feedback
        self.get_logger().info(
            f'Spun {feedback.angular_distance_traveled:.2f} rad'
        )
    
    def goal_response_callback(self, future):
        goal_handle = future.result()
        if not goal_handle.accepted:
            self.get_logger().error('Goal rejected')
            return
        
        # Wait for result
        result_future = goal_handle.get_result_async()
        result_future.add_done_callback(self.result_callback)
    
    def result_callback(self, future):
        result = future.result().result
        self.get_logger().info('Spin recovery complete!')

def main():
    rclpy.init()
    node = RecoveryClient()
    
    # Spin 180 degrees
    node.send_spin_goal(____)  # math.pi
    
    rclpy.spin(node)
    rclpy.shutdown()

if __name__ == '__main__':
    main()

Custom Recovery Plugin (C++)

// my_recovery_behavior.hpp
#include "nav2_core/recovery.hpp"
#include "rclcpp/rclcpp.hpp"
#include "geometry_msgs/msg/twist.hpp"

namespace my_recovery {

class MyRecoveryBehavior : public nav2_core::Recovery {
public:
    void configure(
        const rclcpp_lifecycle::LifecycleNode::WeakPtr & parent,
        const std::string & name,
        std::shared_ptr tf,
        std::shared_ptr costmap_sub,
        std::shared_ptr footprint_sub
    ) override {
        node_ = parent;
        name_ = name;
        
        // Create velocity publisher
        auto node = node_.lock();
        vel_pub_ = node->create_publisher(
            "cmd_vel", 10
        );
        
        // Get parameters
        node->get_parameter("recovery_duration", duration_);
    }
    
    Status cycleBehavior() override {
        auto node = node_.lock();
        
        // Execute recovery action
        geometry_msgs::msg::Twist cmd_vel;
        cmd_vel.linear.x = 0.0;
        cmd_vel.angular.z = ____;  // 0.5 rad/s (turn slowly)
        
        vel_pub_->publish(cmd_vel);
        
        // Check if recovery complete
        if (elapsed_time() > duration_) {
            return Status::SUCCEEDED;
        }
        
        return Status::RUNNING;
    }
    
    void cleanup() override {
        vel_pub_.reset();
    }
    
    void activate() override {}
    void deactivate() override {}

private:
    rclcpp_lifecycle::LifecycleNode::WeakPtr node_;
    std::string name_;
    rclcpp::Publisher::SharedPtr vel_pub_;
    double duration_;
};

} // namespace

// Export plugin
#include "pluginlib/class_list_macros.hpp"
PLUGINLIB_EXPORT_CLASS(
    my_recovery::MyRecoveryBehavior,
    nav2_core::Recovery
)

Common Failures & Fix Patterns

Failure 1: Recovery Causes Collision

Symptom: Spin or backup recovery crashes robot into obstacle.

Root Cause: Collision checking disabled or costmap outdated.

Fix: Enable collision checking and verify costmap:

spin:
  ros__parameters:
    enable_collision_checking: ____  # true (always enable)
    simulate_ahead_time: 2.0  # Check full trajectory

backup:
  ros__parameters:
    enable_collision_checking: true
    backup_speed: 0.05  # Slow speed for safety
# Verify costmap has obstacles
ros2 topic echo /local_costmap/costmap --once
# Check that lethal_cost == 254 for obstacles

# Visualize costmap in RViz
rviz2 -d nav2_default_view.rviz
# Add Map display, topic: /local_costmap/costmap

Failure 2: Infinite Recovery Loop

Symptom: Robot spins, backs up, waits repeatedly without making progress.

Root Cause: Recovery doesn't resolve underlying issue, retry count too high.

Fix: Limit retries and add stuck detection:

<RecoveryNode number_of_retries="____" name="LimitedRecovery">  <!-- 3 (not 10) -->
  <FollowPath path="{path}"/>
  
  <SequenceStar>
    <ClearCostmap/>
    <Spin spin_dist="1.57"/>
    
    <!-- Add stuck condition -->
    <IsStuck/>  <!-- If true, FAIL recovery -->
  </SequenceStar>
</RecoveryNode>

<!-- Fallback to abort mission -->
<Fallback>
  <RecoveryNode .../>
  <AbortMission/>  <!-- Don't loop forever -->
</Fallback>

Failure 3: Recovery Action Times Out

Symptom: Spin/backup action aborts with "Action server timeout".

Root Cause: Slow velocity, large distance, or collision prevents completion.

Fix: Increase time allowance or reduce distance:

backup:
  ros__parameters:
    time_allowance: ____  # 10.0 (increase from default 5.0)
    backup_dist: 0.2  # Reduce distance if needed
# Check if robot is moving during recovery
ros2 topic echo /cmd_vel
# Should see non-zero velocities

# If robot not moving, check:
# 1. Controller enabled?
# 2. E-stop pressed?
# 3. Wheel encoders publishing?

Failure Injection Drill

Exercise: Disable Collision Checking

Force backup recovery to collide with obstacle:

# Intentional error
backup:
  ros__parameters:
    enable_collision_checking: ____  # false (dangerous!)
    backup_dist: 2.0  # Large distance
    backup_speed: 0.2  # Fast speed

Setup: Place obstacle behind robot.

Trigger recovery:

ros2 action send_goal /backup nav2_msgs/action/BackUp \
  "{target: {x: -2.0, y: 0.0, z: 0.0}}"

Observe:

  • Robot backs into obstacle
  • Wheel encoders show slippage
  • Action completes successfully (unaware of collision)
  • Localization error increases (due to slippage)

Detection:

# Monitor motor currents (if available)
ros2 topic echo /motor_status
# High current indicates collision

# Check localization jump
ros2 topic echo /amcl_pose
# Compare pose before/after backup

Recovery:

# Fix config
backup:
  ros__parameters:
    enable_collision_checking: true  # Always enable!
    simulate_ahead_time: 2.0
Warning:

Always enable collision checking for recovery behaviors. The performance cost (10-20ms) is negligible compared to collision damage.

Stretch Goal

Build advanced recovery systems:

  1. Create mission-specific recovery: "DockingRecovery" with approach re-attempt, "ElevatorRecovery" with door detection
  2. Implement probabilistic recovery selection: use ML to choose most-likely-to-succeed recovery based on failure mode
  3. Build human-in-the-loop recovery: request teleoperation assistance when autonomous recovery exhausted
  4. Add recovery cost estimation: prefer fast recoveries (clear costmap) before slow ones (human help)
  5. Create recovery analytics dashboard: track success rates, identify failure patterns
  6. Implement recovery learning: robot learns which recoveries work in which locations
  7. Build recovery simulator: test recovery sequences offline before deployment
