Services

Request/response patterns for synchronous robot operations.

CPU-Light Required

What / Why / When

What

Services implement synchronous request/response communication. A client sends a request and blocks until the server returns a response. Unlike topics, service calls are one-to-one and short-lived. ROS2 provides many built-in services for system configuration and introspection.

Why

Services are ideal for triggering operations that complete quickly and return results. They guarantee delivery and provide clear success/failure feedback. Use them for commands like "capture image," "reset odometry," or "compute path" where you need immediate confirmation.

When

Use services for stateless operations under 1 second. For long-running tasks like navigation goals, use actions instead. Avoid services in tight control loops since blocking calls disrupt real-time performance. If many clients need to call simultaneously, consider queuing or rate-limiting on the server side.

  {
    response->sum = request->____ + request->____;
    RCLCPP_INFO(this->get_logger(), "Request: %ld + %ld = %ld",

Common Failures & Fix Patterns

Failure 1: Service Not Found

Symptom: Client throws "service not available" after timeout.

Root Cause: Server not running, wrong service name, or namespace mismatch.

Fix: Verify server is running and service name matches exactly:

ros2 service list  # Check actual service name
ros2 node info /add_two_ints_server  # Verify server node is alive

Failure 2: Callback Crashes Server

Symptom: Server node exits when handling request with exception.

Root Cause: Unhandled exception in service callback (division by zero, null pointer, etc.).

Fix: Add try-catch blocks in callback:

void handle_divide(Request request, Response response) {
  try {
    if (request->divisor == 0) {
      RCLCPP_ERROR(this->get_logger(), "Division by zero");
      response->success = false;
      return;
    }
    response->quotient = request->dividend / request->divisor;
    response->success = true;
  } catch (const std::exception& e) {
    RCLCPP_ERROR(this->get_logger(), "Exception: %s", e.what());
    response->success = false;
  }
}

Failure 3: Client Hangs Indefinitely

Symptom: spin_until_future_complete never returns.

Root Cause: Server processing stalled or client didn't check service availability first.

Fix: Always check wait_for_service() before calling:

if not self.client.wait_for_service(timeout_sec=5.0):
    self.get_logger().error('Service not available after 5 seconds')
    return None
future = self.client.call_async(request)

Failure Injection Drill

Exercise: Timeout Handling

Create a slow server that delays responses:

import time

def handle_request(self, request, response):
    self.get_logger().info('Processing request...')
    time.sleep(10)  # Simulate slow operation
    response.result = request.input * 2
    return response

Detection: Client with 5-second timeout:

if not self.client.wait_for_service(timeout_sec=5.0):
    self.get_logger().error('Service timeout')
    return

future = self.client.call_async(request)
rclpy.spin_until_future_complete(self, future, timeout_sec=5.0)

if not future.done():
    self.get_logger().error('Request timed out')
else:
    result = future.result()

Recovery: Implement retry logic or increase timeout for known slow operations.

Warning:

Service callbacks run in the same thread as spin by default. Long-running callbacks block other node operations. Use MultiThreadedExecutor for concurrent requests.
