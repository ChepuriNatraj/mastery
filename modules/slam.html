SLAM Toolbox

Build maps in real-time with graph-based SLAM and loop closure.

Mapping Localization

What / Why / When

What

SLAM Toolbox is a graph-based SLAM system implementing Karto SLAM algorithm with modern optimizations. It builds 2D occupancy grid maps from laser scans while simultaneously localizing the robot. Uses pose graph optimization with loop closure detection to correct accumulated drift. Supports synchronous (real-time) and asynchronous (post-processing) modes. Provides map serialization, lifelong mapping, and localization-only mode for pre-built maps.

Why

Traditional SLAM approaches (GMapping, Hector) struggle with large environments due to drift accumulation. SLAM Toolbox uses graph optimization to maintain global consistency even in multi-kilometer maps. Loop closure automatically detects when robot revisits locations and corrects the entire map trajectory. This enables accurate large-scale mapping without manual intervention. The toolbox integrates seamlessly with Nav2 for autonomous navigation.

When

Use SLAM Toolbox for initial map creation in unknown environments. Once map is built, switch to localization mode (with AMCL) for production navigation. SLAM is computationally expensive - consider using asynchronous mode for offline map building from bag files. Requires good odometry and distinctive features in lidar scans. Featureless environments (long hallways, warehouses) may cause tracking failures. Always validate maps manually before deployment.

10-Minute Drill

Launch SLAM Mapping

Start real-time SLAM with default parameters:

# Install SLAM Toolbox
sudo apt install ros-humble-slam-toolbox

# Launch online synchronous SLAM (real-time)
ros2 launch slam_toolbox online_sync_launch.py \
    use_sim_time:=____ \  # true if using Gazebo/bag
    slam_params_file:=/path/to/mapper_params_online_sync.yaml

# Drive robot around to build map
ros2 topic pub /cmd_vel geometry_msgs/msg/Twist "{linear: {x: 0.3}, angular: {z: 0.0}}"

# Visualize in RViz
rviz2 -d `ros2 pkg prefix slam_toolbox`/share/slam_toolbox/rviz/slam_toolbox_default.rviz

    # SLAM Algorithm Parameters
    resolution: ____  # 0.05 (5cm per pixel)

    minimum_travel_distance: ____  # 0.5 (meters before adding scan)
    minimum_travel_heading: 0.5  # radians before adding scan
    scan_buffer_size: 10
    scan_buffer_maximum_scan_distance: 10.0
    
    # Loop Closure
    loop_search_maximum_distance: ____  # 3.0 meters
    do_loop_closing: true

    correlation_search_space_resolution: 0.01

Save Map

Serialize map to file:

# Save map while SLAM is running
ros2 service call /slam_toolbox/save_map slam_toolbox/srv/SaveMap "{name: {data: '/path/to/mymap'}}"

# This creates two files:
# - mymap.yaml (metadata)
# - mymap.pgm (occupancy grid image)

# OR use map_saver from nav2
ros2 run nav2_map_server map_saver_cli -f mymap

Concept ↔ Code (SLAM Modes)

Mapping vs Localization Mode

Mode Use Case Map Updates CPU Usage
mapping Build new map, explore unknown areas Continuous updates, loop closure High (optimization running)
localization Navigate in known map, production use No map changes, only pose estimation Low (no optimization)
lifelong Long-term mapping with dynamic changes Incremental map updates over time Medium

Localization Mode Configuration

# localization_params.yaml
slam_toolbox:
  ros__parameters:
    mode: localization  # Changed from mapping
    
    # Load pre-built map
    map_file_name: /path/to/mymap  # .posegraph file
    map_start_pose: [0.0, 0.0, 0.0]  # x, y, theta
    
    # Localization-specific
    do_loop_closing: false  # Don't modify map
    minimum_travel_distance: 0.2  # More frequent updates
    
    # Still publishes map->odom transform

Asynchronous SLAM (Offline Processing)

# Process bag file offline for high-quality map
ros2 launch slam_toolbox online_async_launch.py \
    use_sim_time:=true \
    slam_params_file:=mapper_params_online_async.yaml

# Play bag (slower than real-time for processing)
ros2 bag play my_mapping_run.bag --clock --rate 0.5

# Async mode runs optimization in background thread
# Better for large maps or constrained hardware

Launch File Integration

from launch import LaunchDescription
from launch.actions import DeclareLaunchArgument
from launch.substitutions import LaunchConfiguration
from launch_ros.actions import Node
from ament_index_python.packages import get_package_share_directory
import os

def generate_launch_description():
    pkg_share = get_package_share_directory('my_robot_slam')
    
    slam_params_file = LaunchConfiguration('slam_params_file')
    use_sim_time = LaunchConfiguration('use_sim_time')
    
    declare_slam_params_cmd = DeclareLaunchArgument(
        'slam_params_file',
        default_value=os.path.join(pkg_share, 'config', 'mapper_params.yaml'),
        description='Full path to slam parameters file'
    )
    
    declare_use_sim_time_cmd = DeclareLaunchArgument(
        'use_sim_time',
        default_value='false',
        description='Use simulation clock if true'
    )
    
    slam_node = Node(
        package='slam_toolbox',
        executable='____',  # sync_slam_toolbox_node or async_slam_toolbox_node
        name='slam_toolbox',
        output='screen',
        parameters=[
            slam_params_file,
            {'use_sim_time': ____}  # use_sim_time
        ]
    )
    
    return LaunchDescription([
        declare_slam_params_cmd,
        declare_use_sim_time_cmd,
        slam_node
    ])

Parameters & Config

Critical Parameter Tuning

Parameter Effect Increase When
minimum_travel_distance Distance before adding scan to graph Map too dense, performance issues
loop_search_maximum_distance How far to search for loop closures Missing obvious loops in large rooms
resolution Map grid cell size Need finer details (small obstacles)
max_laser_range Ignore scans beyond this distance Lidar has longer useful range

Performance Tuning

# For resource-constrained platforms (Raspberry Pi, Jetson)
slam_toolbox:
  ros__parameters:
    # Reduce graph density
    minimum_travel_distance: 1.0  # Larger gaps between nodes
    minimum_travel_heading: 1.0
    
    # Less frequent optimization
    optimize_every_n_nodes: 50  # Default 20
    
    # Limit search space
    loop_search_maximum_distance: 5.0
    correlation_search_space_dimension: 0.3
    
    # Lower resolution map
    resolution: 0.1  # 10cm instead of 5cm
    
    # Reduce update rate
    transform_publish_period: 0.05  # 20Hz instead of 50Hz

Common Failures & Fix Patterns

Failure 1: Map Drift / Poor Odometry

Symptom: Map becomes inconsistent, walls appear duplicated or misaligned, loop closures fail.

Root Cause: Wheel odometry drifts too much between scans, SLAM can't correlate measurements.

Fix: Improve odometry or tune correlation parameters:

# Tighten scan correlation
correlation_search_space_dimension: 0.3  # Reduce from 0.5
correlation_search_space_resolution: 0.005  # Increase precision

# Add scans more frequently
minimum_travel_distance: 0.3  # Reduce from 0.5
minimum_travel_heading: 0.3

# OR improve odometry:
# - Calibrate wheel diameter and base width
# - Add IMU fusion with robot_localization
# - Use visual odometry as supplement

Failure 2: False Loop Closures

Symptom: Map suddenly warps or folds on itself incorrectly.

Root Cause: SLAM incorrectly matches current scan to old location (e.g., similar-looking hallways).

Fix: Increase loop closure thresholds:

# More conservative loop closure
loop_match_minimum_response_coarse: 0.5  # Increase from 0.35
loop_match_minimum_response_fine: 0.45  # Increase from 0.1
loop_match_minimum_chain_size: 15  # Increase from 10

# Reduce search distance to avoid distant false matches
loop_search_maximum_distance: 4.0  # Reduce from 8.0

# If problem persists, disable loop closure temporarily:
do_loop_closing: false

Failure 3: Missing map→odom Transform

Symptom: "Could not transform laser scan into map frame" errors, RViz shows no map.

Root Cause: Frame configuration mismatch or SLAM not initialized.

Fix: Verify frame names and TF tree:

# Check TF tree
ros2 run tf2_tools view_frames
# Expected: map -> odom -> base_footprint -> base_link -> laser_frame

# Verify SLAM parameters match robot frames
ros2 param get /slam_toolbox odom_frame
# Should match odometry publisher frame_id

ros2 param get /slam_toolbox base_frame
# Should match robot base frame

# Check scan message header
ros2 topic echo /scan --once | grep frame_id
# Should match laser frame in URDF/TF tree

# If frames don't match, update config:
# mapper_params.yaml
odom_frame: odom  # Match your odom publisher
base_frame: base_footprint  # Match your robot base
# Then restart SLAM node

        # Publish corrupted odometry
        self.pub = self.create_publisher(Odometry, '/odom', 10)

        # Scale linear velocity by 1.5x (simulate wheel slip)
        msg.twist.twist.linear.x *= ____  # 1.5
        
        # Add noise to angular velocity
        msg.twist.twist.angular.z += ____  # 0.1 radians/sec
        
        self.pub.publish(msg)

# Run this node and observe in RViz:
# - Map drift increases
# - Walls appear doubled
# - Loop closures may fail or be false

Detection: Monitor loop closure success rate:

ros2 topic echo /slam_toolbox/feedback
# Look for "Loop closure failed" vs "Loop closure detected"
# High failure rate indicates odometry or feature issues

Recovery: Tune SLAM to tolerate more odometry error:

minimum_travel_distance: 0.2  # Add scans more frequently
correlation_search_space_dimension: 0.7  # Wider search window
correlation_search_space_smear_deviation: 0.2  # More tolerance
Warning:

SLAM Toolbox requires distinctive environment features. Featureless areas (empty rooms, long hallways) cause tracking loss. Add temporary landmarks during mapping if needed.
