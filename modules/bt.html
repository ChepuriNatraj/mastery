Behavior Trees

Compose robust navigation logic with modular, reactive decision-making.

Nav2 AI Logic

What / Why / When

What

Behavior Trees (BTs) are hierarchical state machines for robot decision-making. Nav2 uses BehaviorTree.CPP library with XML-defined trees. BTs consist of control nodes (Sequence, Fallback, Parallel) and leaf nodes (actions, conditions). Each node returns SUCCESS, FAILURE, or RUNNING. Execution flows from root, ticking children based on control logic. Nav2 provides built-in BT nodes: ComputePathToPose, FollowPath, Spin, BackUp, ClearCostmap, etc. Trees are reusable, composable, and modifiable without code changes. Support features: blackboard for data sharing, decorators for retry logic, and subtrees for modularity.

Why

Finite state machines (FSMs) become unwieldy for complex behaviors - adding recovery logic requires many state transitions. BTs provide modularity: add recovery behaviors without restructuring entire tree. Reactive execution: BTs tick continuously, adapting to changing conditions (unlike FSMs stuck in states). Reusable components: Create common subtrees (docking, patrolling) and compose into missions. Visual representation: XML trees are human-readable, easy to debug. BTs enable non-programmers to modify robot behavior through tree editing.

When

Use BTs for all Nav2 navigation - they orchestrate planning, control, and recovery. Customize default trees for mission-specific logic (warehouse patrol, delivery, inspection). Create separate BT files for different missions, loaded dynamically. BTs are declarative but can be verbose - simple behaviors may be clearer as code. Always validate BT syntax before deployment - XML errors cause runtime crashes. Use Groot for visual BT editing and monitoring during execution.

Basic BT Structure

Create simple navigation behavior tree:

<?xml version="1.0"?>
<root main_tree_to_execute="MainTree">
  <BehaviorTree ID="MainTree">
    
    <!-- Sequence: Execute children in order, fail if any fails -->
    <Sequence name="NavigateWithRecovery">
      
      <!-- Compute global path -->
      <ComputePathToPose 
        goal="{goal}" 
        path="{path}"
        planner_id="____"/>  <!-- GridBased -->
      
      <!-- Follow path with recovery -->
      <RecoveryNode number_of_retries="3" name="FollowWithRecovery">
        
        <!-- Main behavior -->
        <FollowPath 
          path="{path}" 
          controller_id="FollowPath"/>
        
        <!-- Recovery on failure -->
        <SequenceStar name="RecoveryActions">
          <ClearEntireCostmap 
            name="ClearLocalCostmap"
            service_name="____"/>  <!-- local_costmap/clear_entirely_local_costmap -->
          <Spin spin_dist="1.57"/>  <!-- 90 degrees -->
        </SequenceStar>
        
      </RecoveryNode>
      
    </Sequence>
    
  </BehaviorTree>
</root>

Control Node Types

Node Behavior Use Case
Sequence Execute children left-to-right. Fail on first failure. Required steps (plan → execute → verify)
SequenceStar Like Sequence but restart from beginning on retry Recovery sequences
Fallback Try children until one succeeds Alternative strategies (Plan A, Plan B, Plan C)
Parallel Execute children simultaneously Monitor + execute (e.g., time limit + action)
RecoveryNode Retry main child, run recovery on failure Action with fallback (navigate with spin recovery)
PipelineSequence Like Sequence but continue from last running child Replanning (keep executing, replan in parallel)

Concept ↔ Code (BT Execution)

BT Tick Execution Flow

# Conceptual BT execution
class BehaviorTree:
    def __init__(self, root_node):
        self.root = root_node
        self.blackboard = {}  # Shared data
    
    def tick(self):
        # Execute tree from root
        return self.root.tick(self.blackboard)

class SequenceNode:
    def __init__(self, children):
        self.children = children
        self.current_child = 0
    
    def tick(self, blackboard):
        # Execute children in order
        while self.current_child < len(self.children):
            status = self.children[self.current_child].tick(blackboard)
            
            if status == Status.RUNNING:
                return Status.____  # RUNNING
            elif status == Status.FAILURE:
                self.current_child = 0  # Reset
                return Status.FAILURE
            else:  # SUCCESS
                self.current_child += 1
        
        self.current_child = 0  # Reset for next execution
        return Status.SUCCESS

class RecoveryNode:
    def __init__(self, main_child, recovery_child, retries):
        self.main = main_child
        self.recovery = recovery_child
        self.max_retries = retries
        self.retry_count = 0
    
    def tick(self, blackboard):
        # Try main behavior
        status = self.main.tick(blackboard)
        
        if status == Status.RUNNING or status == Status.SUCCESS:
            self.retry_count = 0
            return status
        
        # Main failed - try recovery
        if self.retry_count < self.max_retries:
            self.retry_count += 1
            recovery_status = self.recovery.tick(blackboard)
            
            if recovery_status == Status.SUCCESS:
                return Status.____  # RUNNING (retry main)
            else:
                return Status.RUNNING  # Keep trying recovery
        
        # Out of retries
        return Status.FAILURE

Blackboard Data Sharing

<!-- Write to blackboard -->
<ComputePathToPose 
  goal="{goal}"        <!-- INPUT: Read from blackboard key "goal" -->
  path="{path}"/>      <!-- OUTPUT: Write to blackboard key "path" -->

<!-- Read from blackboard -->
<FollowPath 
  path="{path}"/>      <!-- INPUT: Read "path" written above -->

<!-- Blackboard in C++: -->
// Write
config().blackboard->set("path", computed_path);

// Read
nav_msgs::msg::Path path;
config().blackboard->get("path", path);

Custom BT Node Plugin

// my_bt_node.hpp
#include "behaviortree_cpp_v3/action_node.h"
#include "rclcpp/rclcpp.hpp"

namespace my_bt_nodes {

class MyCustomNode : public BT::SyncActionNode {
public:
    MyCustomNode(
        const std::string & name,
        const BT::NodeConfiguration & config,
        rclcpp::Node::SharedPtr node)
    : SyncActionNode(name, config), node_(node) {}
    
    static BT::PortsList providedPorts() {
        return {
            BT::InputPort("input_param"),
            BT::OutputPort("output_result")
        };
    }
    
    BT::NodeStatus tick() override {
        // Get input from blackboard
        auto input = getInput("input_param");
        if (!input) {
            return BT::NodeStatus::____;  // FAILURE
        }
        
        // Your logic here
        int result = do_computation(input.value());
        
        // Write output to blackboard
        setOutput("output_result", result);
        
        return BT::NodeStatus::SUCCESS;
    }

private:
    rclcpp::Node::SharedPtr node_;
};

} // namespace

// Register plugin
#include "behaviortree_cpp_v3/bt_factory.h"
BT_REGISTER_NODES(factory) {
    factory.registerNodeType("MyCustomNode");
}

Common Failures & Fix Patterns

Failure 1: BT XML Parsing Error

Symptom: bt_navigator fails to start with "Failed to parse BT XML" error.

Root Cause: Invalid XML syntax, typo in node name, or missing required port.

Fix: Validate XML and check node definitions:

# Check XML syntax
xmllint --noout /path/to/my_bt.xml
# Expected: No errors

# Common mistakes:
# 1. Missing closing tag
<Sequence>
  <ComputePathToPose .../>
# </Sequence>  <!-- MISSING -->

# 2. Typo in node name
<ComputePathToPose .../>  # Correct
<ComputPathToPose .../>   # WRONG (missing 'e')

# 3. Missing required port
<ComputePathToPose goal="{goal}"/>  # MISSING path="{path}"

# Fix: Add all required ports
<ComputePathToPose goal="{goal}" path="{____}"/>  # path

Failure 2: BT Returns FAILURE Immediately

Symptom: Navigation aborts instantly without attempting to plan/execute.

Root Cause: Condition node failing early or blackboard data missing.

Fix: Add logging and check blackboard:

<!-- Add logging nodes for debugging -->
<Sequence>
  <AlwaysSuccess name="LogStart">
    <!-- Prints to log when executed -->
  </AlwaysSuccess>
  
  <ComputePathToPose goal="{goal}" path="{path}"/>
  
  <AlwaysSuccess name="LogPathComputed"/>
  
  <FollowPath path="{path}"/>
</Sequence>

<!-- Check if goal is set in blackboard -->
# If goal is null/empty, ComputePathToPose fails immediately

Failure 3: Infinite Recovery Loop

Symptom: Robot repeatedly executes recovery behaviors, never makes progress.

Root Cause: Recovery doesn't fix underlying issue, or retry count too high.

Fix: Limit retries and add failure detection:

<RecoveryNode number_of_retries="____" name="LimitedRecovery">  <!-- 3 (not 99) -->
  <FollowPath path="{path}"/>
  
  <!-- Recovery actions -->
  <SequenceStar>
    <ClearCostmap/>
    <Spin spin_dist="1.57"/>
    
    <!-- Add condition to break loop -->
    <IsStuck/>  <!-- If stuck, FAIL recovery -->
  </SequenceStar>
</RecoveryNode>

<!-- After RecoveryNode fails, use Fallback for alternative -->
<Fallback>
  <RecoveryNode .../>  <!-- Try navigation -->
  <SubTree ID="AbortAndReturnHome"/>  <!-- Fallback plan -->
</Fallback>

Failure Injection Drill

Exercise: Break Blackboard Data Flow

Mismatch blackboard keys to see execution failure:

<!-- Intentional error -->
<Sequence>
  <!-- Writes to "path" -->
  <ComputePathToPose goal="{goal}" path="{path}"/>
  
  <!-- Tries to read "wrong_key" (doesn't exist) -->
  <FollowPath path="{____}"/>  <!-- wrong_key instead of path -->
</Sequence>

<!-- Observe:
- ComputePathToPose succeeds, writes path
- FollowPath fails immediately with "Missing required input"
- Sequence returns FAILURE
- Navigation aborts
-->

Detection: Enable BT logging:

# Set log level to DEBUG
ros2 run nav2_bt_navigator bt_navigator --ros-args \
  --log-level DEBUG

# Logs show:
# "Port [path] not found in blackboard"

Recovery: Match blackboard keys:

<FollowPath path="{path}"/>  <!-- Corrected -->
Warning:

BT nodes execute at 10Hz by default (bt_loop_duration: 10ms). Don't add CPU-intensive operations directly in BT nodes - use action servers called from BT instead.
