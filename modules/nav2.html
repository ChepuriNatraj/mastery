Nav2 Stack

Orchestrate autonomous navigation with behavior trees and lifecycle management.

Nav2 Autonomous

What / Why / When

What

Nav2 (Navigation2) is the ROS2 navigation framework providing autonomous mobile robot navigation. It orchestrates localization (AMCL), global/local planning, control, costmap management, and recovery behaviors. Uses behavior trees for decision-making and lifecycle nodes for coordinated state management. Supports waypoint following, dynamic replanning, keepout zones, and plugin-based extensibility. Includes built-in planners (NavFn, Smac), controllers (DWB, RPP), and behavior tree nodes for complex navigation logic.

Why

Building navigation from scratch is complex - path planning, obstacle avoidance, recovery, and coordination require thousands of lines of code. Nav2 provides battle-tested components with rich configuration options. Behavior trees enable flexible mission planning without code changes. Lifecycle management ensures graceful startup/shutdown and error handling. Plugin architecture allows custom algorithms without forking. Nav2 is production-ready, used in commercial robots, and actively maintained by Open Navigation LLC.

When

Use Nav2 for any autonomous mobile robot requiring point-to-point navigation. Works in indoor/outdoor environments with 2D maps. Not suitable for 3D navigation (drones, humanoids) or non-holonomic constraints beyond differential/ackermann drives. Requires accurate localization and odometry. Nav2 is computationally intensive - ensure robot has sufficient CPU (recommended: 4+ cores). Always test in simulation before hardware deployment.

Nav2 Components Architecture

Nav2 Stack Components:

bt_navigator ──────→ Executes behavior tree logic
  ├─→ Calls planner_server for global path
  ├─→ Calls controller_server for local control
  └─→ Calls recovery behaviors on failure

planner_server ────→ Global path planning (NavFn, Smac, etc.)
controller_server ─→ Local trajectory control (DWB, RPP, etc.)
recoveries_server ─→ Recovery behaviors (spin, backup, wait)
waypoint_follower ─→ Multi-waypoint missions
behavior_server ───→ High-level behaviors (dock, undock)

amcl ──────────────→ Localization (separate package)
lifecycle_manager ─→ Coordinates node startup/shutdown

costmap_2d ────────→ Global and local costmaps (shared library)

Send Goal from Code

Navigate programmatically with action client:

import rclpy
from rclpy.node import Node
from rclpy.action import ActionClient
from nav2_msgs.action import NavigateToPose
from geometry_msgs.msg import PoseStamped

class NavGoalSender(Node):
    def __init__(self):
        super().__init__('nav_goal_sender')
        
        self.action_client = ActionClient(
            self,
            NavigateToPose,
            '____'  # /navigate_to_pose
        )
        
    def send_goal(self, x, y, yaw=0.0):
        goal_msg = NavigateToPose.Goal()
        
        goal_msg.pose.header.frame_id = '____'  # map
        goal_msg.pose.header.stamp = self.get_clock().now().to_msg()
        
        goal_msg.pose.pose.position.x = x
        goal_msg.pose.pose.position.y = ____  # y
        goal_msg.pose.pose.position.z = 0.0
        
        # Convert yaw to quaternion
        from tf_transformations import quaternion_from_euler
        quat = quaternion_from_euler(0, 0, yaw)
        goal_msg.pose.pose.orientation.x = quat[0]
        goal_msg.pose.pose.orientation.y = quat[1]
        goal_msg.pose.pose.orientation.z = quat[2]
        goal_msg.pose.pose.orientation.w = ____  # quat[3]
        
        self.get_logger().info(f'Sending goal: x={x}, y={y}')
        
        # Wait for action server
        self.action_client.wait_for_server()
        
        # Send goal with feedback callback
        send_goal_future = self.action_client.send_goal_async(
            goal_msg,
            feedback_callback=self.feedback_callback
        )
        send_goal_future.add_done_callback(self.goal_response_callback)
    
    def goal_response_callback(self, future):
        goal_handle = future.result()
        if not goal_handle.accepted:
            self.get_logger().warn('Goal rejected')
            return
        
        self.get_logger().info('Goal accepted')
        result_future = goal_handle.get_result_async()
        result_future.add_done_callback(self.result_callback)
    
    def result_callback(self, future):
        result = future.result().result
        self.get_logger().info(f'Navigation result: {result}')
    
    def feedback_callback(self, feedback_msg):
        feedback = feedback_msg.feedback
        # Distance remaining, estimated time, etc.
        pass

def main():
    rclpy.init()
    node = NavGoalSender()
    node.send_goal(5.0, 3.0, 1.57)  # Go to (5, 3) facing 90°
    rclpy.spin(node)
    rclpy.shutdown()

Concept ↔ Code (Core Workflow)

Navigation Execution Flow

1. User sends NavigateToPose goal to bt_navigator

2. bt_navigator executes behavior tree:
   a. ComputePathToPose: Call planner_server for global path
      → Returns path as nav_msgs/Path
   
   b. FollowPath: Send path to controller_server
      → Controller computes cmd_vel at high rate (20-50 Hz)
      → Robot moves following local trajectory
   
   c. Monitor progress:
      - If obstacle blocks path → Replan (call planner again)
      - If stuck → Trigger recovery (spin, backup, clear costmap)
      - If timeout → Abort mission
   
   d. Goal reached when within xy_goal_tolerance and yaw_goal_tolerance

3. bt_navigator returns SUCCESS/FAILURE

Behavior Tree Structure

<!-- Simplified BT for NavigateToPose -->
<root main_tree_to_execute="MainTree">
  <BehaviorTree ID="MainTree">
    <PipelineSequence name="NavigateWithReplanning">
      
      <!-- Try to compute and follow path -->
      <RecoveryNode number_of_retries="6" name="NavigateRecovery">
        <PipelineSequence name="NavigateWithReplanningAndRecovery">
          
          <!-- Compute path -->
          <ComputePathToPose goal="{goal}" path="{path}" 
                             planner_id="GridBased"/>
          
          <!-- Follow path with replanning -->
          <RecoveryNode number_of_retries="____">  <!-- 3 -->
            <FollowPath path="{path}" controller_id="FollowPath"/>
            
            <!-- On failure, clear costmap and retry -->
            <SequenceStar name="RecoveryActions">
              <ClearEntireCostmap name="ClearGlobalCostmap" 
                                  service_name="global_costmap/clear_entirely_global_costmap"/>
              <ClearEntireCostmap name="ClearLocalCostmap"
                                  service_name="local_costmap/clear_entirely_local_costmap"/>
            </SequenceStar>
          </RecoveryNode>
          
        </PipelineSequence>
        
        <!-- Fallback recovery behaviors -->
        <SequenceStar name="RecoveryActions">
          <Spin spin_dist="____"/>  <!-- 1.57 radians (90°) -->
          <Wait wait_duration="5"/>
          <BackUp backup_dist="0.3" backup_speed="0.05"/>
        </SequenceStar>
        
      </RecoveryNode>
      
    </PipelineSequence>
  </BehaviorTree>
</root>

Waypoint Following

# Follow multiple waypoints in sequence
from nav2_msgs.action import FollowWaypoints
from geometry_msgs.msg import PoseStamped

class WaypointFollower(Node):
    def __init__(self):
        super().__init__('waypoint_follower')
        self.action_client = ActionClient(
            self, FollowWaypoints, '/follow_waypoints')
    
    def follow_waypoints(self, waypoints):
        goal_msg = FollowWaypoints.Goal()
        
        for (x, y) in waypoints:
            pose = PoseStamped()
            pose.header.frame_id = 'map'
            pose.header.stamp = self.get_clock().now().to_msg()
            pose.pose.position.x = x
            pose.pose.position.y = y
            pose.pose.orientation.w = 1.0
            goal_msg.poses.append(pose)
        
        self.action_client.wait_for_server()
        return self.action_client.send_goal_async(goal_msg)

# Usage:
node = WaypointFollower()
node.follow_waypoints([(1.0, 1.0), (2.0, 3.0), (4.0, 2.0)])

    # Plugin names (must match planner/controller server configs)

    # Goal checker
    goal_blackboard_id: "goal"
    goals_blackboard_id: "goals"

    # Behavior tree options

Common Failures & Fix Patterns

Failure 1: Goal Rejected - "Too Far"

Symptom: bt_navigator immediately rejects goal with "Goal is too far" error.

Root Cause: Goal outside global costmap bounds or planner max range.

Fix: Increase costmap size or planner limits:

# Increase global costmap size
global_costmap:
  ros__parameters:
    width: ____  # 50 (meters, was maybe 20)
    height: 50

# OR increase planner max search distance
planner_server:
  ros__parameters:
    GridBased:
      plugin: "nav2_navfn_planner/NavfnPlanner"
      use_final_approach_orientation: false
      # NavFn auto-uses entire costmap, but Smac has limits:
      
    # If using SmacPlanner:
    SmacHybrid:
      max_planning_time: 5.0  # seconds
      allow_unknown: false

Failure 2: Robot Oscillates / Doesn't Reach Goal

Symptom: Robot moves back and forth near goal but never completes, or aborts with "timeout".

Root Cause: Goal tolerance too strict, controller can't achieve precise pose.

Fix: Relax tolerances:

controller_server:
  ros__parameters:
    FollowPath:
      xy_goal_tolerance: 0.5  # Increase from 0.25
      yaw_goal_tolerance: 0.5  # Increase from 0.25
      
      # Stop checking xy after first reaching it
      latch_xy_goal_tolerance: ____  # true
      
      # Slow down near goal for precision
      approach_velocity_scaling_dist: 1.0
      min_approach_linear_velocity: 0.05

Failure 3: Lifecycle Manager Timeout

Symptom: "Failed to bring up nodes" error on launch, some nodes not activated.

Root Cause: Node takes too long to configure/activate or crash during transition.

Fix: Check individual node status and increase timeout:

# Manually test lifecycle transitions
ros2 lifecycle set /controller_server configure
# If this times out, controller_server has config issue

# Check logs for specific error
ros2 run nav2_lifecycle_manager lifecycle_manager --ros-args \
  -p bond_timeout:=____  # 10.0 (increase from 4.0)

# OR disable bond (not recommended for production)
lifecycle_manager:
  ros__parameters:
    bond_timeout: 0.0  # Disable connection monitoring

Failure Injection Drill

Exercise: Crash Recovery Server and Observe

Kill recovery server to see Nav2 fault handling:

# Terminal 1: Launch Nav2
ros2 launch nav2_bringup navigation_launch.py map:=mymap.yaml

# Terminal 2: Send navigation goal
ros2 action send_goal /navigate_to_pose nav2_msgs/action/NavigateToPose \
  "{pose: {header: {frame_id: 'map'}, pose: {position: {x: 5.0, y: 3.0}}}}"

# Terminal 3: Kill recoveries_server mid-navigation
ros2 lifecycle set /recoveries_server deactivate
ros2 lifecycle set /recoveries_server ____  # cleanup (shutdown)

# Observe:
# - If robot gets stuck, bt_navigator can't call recovery behaviors
# - Navigation may abort with "recovery server unavailable"
# - Lifecycle manager attempts respawn if configured

Detection: Monitor bond status:

ros2 topic echo /bond
# Shows active bonds between lifecycle_manager and managed nodes
# Missing bond indicates node crash

Recovery: Restart failed node:

# Lifecycle manager auto-restarts if attempt_respawn_reconnection: true
# OR manually restart:
ros2 lifecycle set /recoveries_server configure
ros2 lifecycle set /recoveries_server activate

# Then resend navigation goal
Warning:

Nav2 requires significant CPU and memory. On resource-constrained platforms (Raspberry Pi 3), reduce controller/planner frequencies and use simpler algorithms (NavFn instead of SmacHybrid).
