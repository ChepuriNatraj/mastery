Gazebo Simulation

Spawn robots, test sensors, and iterate without hardware.

Requires Gazebo Needs RViz

What / Why / When

What

Gazebo is a 3D physics simulator for robots. It simulates rigid body dynamics, sensors (lidar, camera, IMU), actuators, and environments. Gazebo integrates with ROS2 via plugins that publish sensor data and subscribe to commands. It uses ODE, Bullet, or Dart physics engines. Gazebo Classic (version 11) and Gazebo (Ignition/Fortress+) are both actively used with ROS2 Humble.

Why

Hardware testing is slow, expensive, and risky during development. Gazebo enables rapid prototyping, algorithm validation, and edge case testing without physical robots. It provides reproducible test scenarios, simulates sensor noise realistically, and supports multi-robot systems. SLAM, navigation, and manipulation can be fully tested in simulation before hardware deployment.

When

Use Gazebo for initial algorithm development, CI/CD testing, and scenarios dangerous for real robots. Always validate sim-to-real transfer with hardware tests. Gazebo physics isn't perfect - contact forces, friction, and deformable objects have limitations. Use for high-level behavior testing, not precise dynamics. Always set use_sim_time:=true for all nodes when using Gazebo.

10-Minute Drill

Launch Gazebo with Empty World

Start Gazebo and spawn a robot:

# launch/gazebo_world.launch.py
from launch import LaunchDescription
from launch.actions import IncludeLaunchDescription, ExecuteProcess
from launch.launch_description_sources import PythonLaunchDescriptionSource
from launch_ros.actions import Node
from ament_index_python.packages import get_package_share_directory
import os

def generate_launch_description():
    # Launch Gazebo with empty world
    gazebo_launch = IncludeLaunchDescription(
        PythonLaunchDescriptionSource([
            os.path.join(
                get_package_share_directory('____'),  # gazebo_ros
                'launch',
                'gazebo.launch.py'
            )
        ]),
        launch_arguments={
            'world': os.path.join(
                get_package_share_directory('my_robot_gazebo'),
                'worlds',
                'empty.world'
            ),
            'verbose': '____'  # true or false
        }.items()
    )
    
    # Spawn robot from URDF
    urdf_file = os.path.join(
        get_package_share_directory('my_robot_description'),
        'urdf',
        'robot.urdf'
    )
    
    spawn_entity = Node(
        package='____',  # gazebo_ros
        executable='spawn_entity.py',
        arguments=[
            '-entity', 'my_robot',
            '-file', ____,  # urdf_file
            '-x', '0.0',
            '-y', '0.0',
            '-z', '0.1'
        ],
        output='screen'
    )
    
    return LaunchDescription([
        gazebo_launch,
        spawn_entity
    ])

Concept ↔ Code (Common Plugins)

Laser Scanner Plugin

<gazebo reference="laser_frame">
  <sensor type="ray" name="laser_sensor">
    <always_on>true</always_on>
    <update_rate>10</update_rate>
    <visualize>true</visualize>
    
    <ray>
      <scan>
        <horizontal>
          <samples>360</samples>
          <resolution>1.0</resolution>
          <min_angle>-3.14159</min_angle>
          <max_angle>3.14159</max_angle>
        </horizontal>
      </scan>
      <range>
        <min>0.1</min>
        <max>10.0</max>
        <resolution>0.01</resolution>
      </range>
      <noise>
        <type>gaussian</type>
        <mean>0.0</mean>
        <stddev>0.01</stddev>
      </noise>
    </ray>
    
    <plugin name="laser_controller" filename="libgazebo_ros_ray_sensor.so">
      <ros>
        <remapping>~/out:=scan</remapping>
      </ros>
      <output_type>sensor_msgs/LaserScan</output_type>
      <frame_name>laser_frame</frame_name>
    </plugin>
  </sensor>
</gazebo>

Simulation Time Configuration

# CRITICAL: All nodes must use simulation time
controller:
  ros__parameters:
    use_sim_time: true  # Get time from /clock topic, not system clock

slam_toolbox:
  ros__parameters:
    use_sim_time: true

# Launch argument pattern
ros2 launch my_pkg navigation.launch.py use_sim_time:=true

Common Failures & Fix Patterns

Failure 1: Sensor Topics Not Published

Symptom: Robot spawns in Gazebo but /scan or /camera topics don't exist.

Root Cause: Plugin not properly configured in URDF or sensor link missing collision/inertial.

Fix: Verify plugin block and sensor link completeness:

<!-- Sensor link must have mass and inertia -->
<link name="laser_frame">
  <visual>...</visual>
  <collision>
    <geometry><cylinder radius="0.05" length="0.05"/></geometry>
  </collision>
  <inertial>
    <mass value="0.1"/>
    <inertia ixx="0.001" ixy="0" ixz="0" iyy="0.001" iyz="0" izz="0.001"/>
  </inertial>
</link>

<!-- Plugin must reference correct link -->
<gazebo reference="laser_frame">  <!-- Match link name exactly -->
  <sensor type="ray" name="laser_sensor">
    ...
  </sensor>
</gazebo>

Failure 2: Time Synchronization Errors

Symptom: "Message from [time] is out of bounds" errors, TF lookup failures.

Root Cause: Some nodes use system time while Gazebo publishes simulation time.

Fix: Ensure all nodes use sim time:

# In every launch file when using Gazebo
def generate_launch_description():
    use_sim_time = LaunchConfiguration('use_sim_time', default='true')
    
    return LaunchDescription([
        DeclareLaunchArgument('use_sim_time', default_value='true'),
        
        Node(
            package='my_pkg',
            executable='my_node',
            parameters=[{'use_sim_time': use_sim_time}]
        )
    ])

# Verify /clock is being published
ros2 topic hz /clock
# Expected: ~1000 Hz from Gazebo

Failure 3: Robot Explodes or Oscillates Wildly

Symptom: Robot flies apart, vibrates violently, or falls through ground.

Root Cause: Invalid inertia matrix, incorrect contact parameters, or too-high gains.

Fix: Use realistic physical properties:

<!-- Use proper inertia (positive definite matrix) -->
<inertial>
  <mass value="10.0"/>
  <!-- Calculate from geometry, don't guess -->
  <inertia ixx="0.1" ixy="0.0" ixz="0.0"
           iyy="0.1" iyz="0.0"
           izz="0.1"/>
</inertial>

<!-- Set reasonable contact parameters -->
<gazebo reference="base_link">
  <kp>1000000.0</kp>  <!-- Contact stiffness -->
  <kd>10.0</kd>  <!-- Contact damping -->
  <mu1>0.5</mu1>  <!-- Friction coefficient -->
  <mu2>0.5</mu2>
</gazebo>

<!-- Reduce physics step size if vibration persists -->
<physics type="ode">
  <max_step_size>0.001</max_step_size>
  <real_time_update_rate>1000</real_time_update_rate>
</physics>

Failure Injection Drill

Exercise: Detect Missing Simulation Time

Launch nodes without use_sim_time and observe failures:

# Terminal 1: Launch Gazebo (publishes /clock)
ros2 launch gazebo_ros gazebo.launch.py

# Terminal 2: Launch node WITHOUT sim time parameter
ros2 run my_pkg my_node
# Node uses system time, Gazebo uses sim time

# Terminal 3: Try TF lookup
ros2 run tf2_ros tf2_echo odom base_link
# Expected: Extrapolation errors, "future" warnings

Detection: Check parameter:

ros2 param get /my_node use_sim_time
# If false or not set, node will have time sync issues

Recovery: Restart node with correct parameter:

ros2 run my_pkg my_node --ros-args -p use_sim_time:=true
Warning:

Gazebo runs slower than real-time on resource-constrained machines. Monitor real-time factor in Gazebo GUI (should be ~1.0). Lower physics update rate or simplify world if factor drops below 0.5.
