Controllers

Track global paths with dynamic obstacle avoidance and smooth velocity commands.

Nav2 Control

What / Why / When

What

Controllers (local planners) compute cmd_vel commands to follow global path while avoiding dynamic obstacles. Nav2 provides DWB (Dynamic Window Approach), RPP (Regulated Pure Pursuit), and TEB (Timed Elastic Band) controllers. Controllers run at high frequency (20-50 Hz), use local costmap for real-time obstacle detection, and generate short-horizon trajectories. Each controller scores candidate velocities based on path alignment, obstacle clearance, and goal progress. Best trajectory is selected and corresponding velocity is published. Support features: velocity smoothing, oscillation detection, and path pruning.

Why

Global paths are collision-free at planning time but don't handle dynamic obstacles (people, other robots). Controllers provide reactive obstacle avoidance by continuously replanning local trajectories. They also handle robot dynamics - smoothing accelerations and respecting velocity/acceleration limits. Different controllers suit different robot types: DWB for differential drive, RPP for car-like robots, TEB for minimum-time trajectories. Controllers are critical for safe, smooth navigation - poor tuning causes oscillations, getting stuck, or collisions.

When

Use DWB for general differential drive robots - it's well-tested and configurable. Use RPP for ackermann steering or when simple, smooth following is sufficient. Use TEB when minimum-time trajectories are critical (warehouse robots). Controller choice affects CPU usage: DWB ~5-10% CPU, TEB ~15-25%. Always tune controller gains empirically - simulate in Gazebo first, then test on hardware. Monitor cmd_vel output - sudden velocity changes indicate tuning issues.

Concept ↔ Code (DWB Algorithm)

Dynamic Window Approach

# Conceptual DWB algorithm
def dwb_control(current_vel, goal, obstacles, path):
    # 1. Generate velocity samples within dynamic window
    candidates = []
    for vx in linspace(min_vx, max_vx, vx_samples):
        for vtheta in linspace(min_vtheta, max_vtheta, vtheta_samples):
            # Check if velocity is achievable given acceleration limits
            if is_reachable(current_vel, (vx, vtheta), acc_limits, dt):
                candidates.append((vx, vtheta))
    
    # 2. Simulate trajectory for each candidate
    trajectories = []
    for vel in candidates:
        traj = simulate_trajectory(vel, sim_time)
        
        # Check collision
        if not check_collision(traj, obstacles):
            continue  # Skip unsafe trajectories
        
        trajectories.append((vel, traj))
    
    # 3. Score each trajectory with critics
    best_vel = None
    best_score = -inf
    
    for vel, traj in trajectories:
        score = 0
        score += path_align_critic(traj, path) * ____  # path_align_weight
        score += goal_dist_critic(traj, goal) * goal_dist_weight
        score += obstacle_critic(traj, obstacles) * obstacle_weight
        score -= oscillation_penalty(vel, prev_vel)
        
        if score > best_score:
            best_score = score
            best_vel = vel
    
    # 4. Publish best velocity
    return best_vel  # (vx, vy, vtheta)

Controller Comparison

Controller Approach CPU Smoothness Use Case
DWB Sample velocities, score trajectories Medium Good with tuning General differential drive
RPP Pure pursuit with regulation Low Very smooth Ackermann, simple paths
TEB Elastic band optimization High Optimal but jerky Minimum-time trajectories

DWB Critics Explained

Critic Purpose Increase Weight When
PathAlign Keep heading aligned with path Robot cuts corners
PathDist Stay close to global path Robot deviates from path
GoalAlign Point toward goal Robot approaches at wrong angle
GoalDist Prefer progress toward goal Robot stops short of goal
BaseObstacle Avoid obstacles Robot gets too close to walls
Oscillation Penalize velocity reversals Robot wobbles back/forth
RotateToGoal Rotate in place near goal Need precise final orientation

Common Failures & Fix Patterns

Failure 1: Robot Oscillates Near Goal

Symptom: Robot moves forward/backward repeatedly, never reaching goal.

Root Cause: Goal tolerance too tight or conflicting critic weights.

Fix: Relax tolerance and tune critics:

# Increase goal tolerance
goal_checker:
  xy_goal_tolerance: 0.3  # Was 0.1
  stateful: ____  # true (latch xy goal once reached)

# Reduce oscillation
Oscillation.scale: 1.0  # Add if not present

# Reduce goal approach precision
RotateToGoal.slowing_factor: 3.0  # Slow down sooner

Failure 2: Robot Gets Stuck in Narrow Passage

Symptom: Controller aborts with "No valid trajectories" in doorways/corridors.

Root Cause: All sampled velocities cause predicted collision.

Fix: Increase trajectory resolution and reduce obstacle penalty:

# Finer trajectory simulation
vx_samples: 30  # Was 20
vtheta_samples: 60  # Was 40

# Allow closer approach to obstacles
BaseObstacle.scale: 0.01  # Was 0.02

# Check costmap inflation (may be too large)
inflation_radius: 0.4  # Reduce if robot fits with margin

Failure 3: Robot Ignores Dynamic Obstacles

Symptom: Robot collides with moving people/objects.

Root Cause: Local costmap not updating fast enough or clearing too aggressively.

Fix: Tune local costmap update rate:

local_costmap:
  local_costmap:
    update_frequency: 10.0  # Was 5.0 (update faster)
    
    obstacle_layer:
      clearing: true
      # Don't clear occupied cells immediately
      min_obstacle_height: 0.1  # Ensure detecting obstacles
      obstacle_max_range: 3.0  # See obstacles sooner

# Increase obstacle critic weight
BaseObstacle.scale: ____  # 0.05 (was 0.02)

Warning:

Controller sampling rates (vx_samples, vtheta_samples) directly impact CPU usage. Start with 20x40 samples. Only increase if controller can't find valid trajectories in complex environments.
