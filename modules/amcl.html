AMCL Localization

Track robot pose in known maps with particle filters.

Localization Nav2

What / Why / When

What

AMCL (Adaptive Monte Carlo Localization) estimates robot pose in a known map using laser scans and odometry. It implements a particle filter - maintains a set of weighted hypotheses (particles) representing possible robot poses. Each particle is updated based on odometry motion model and weighted by scan-to-map correlation. Particles converge to most likely pose over time. Adapts particle count dynamically based on localization confidence (KLD-sampling). Publishes map→odom transform to maintain global consistency.

Why

Odometry-only navigation accumulates unbounded error - robot drifts from true position. AMCL corrects this drift by matching sensor observations to map. Unlike EKF localization, particle filters can track multiple hypotheses (useful for ambiguous symmetric environments) and recover from kidnapped robot problem. AMCL is computationally efficient, handles non-Gaussian noise, and integrates seamlessly with Nav2 stack for autonomous navigation.

When

Use AMCL for production navigation in static, pre-mapped environments. Requires accurate 2D map and working odometry. Not suitable for dynamic environments (many moving obstacles), outdoor navigation without clear features, or 3D localization. AMCL needs initial pose estimate - either manual (RViz 2D Pose Estimate) or from known starting position. Kidnapped robot recovery takes time and may fail in featureless areas. Always validate localization quality before autonomous operation.

Concept ↔ Code (Particle Filter)

Particle Filter Algorithm

# Conceptual AMCL particle filter (simplified)
class ParticleFilter:
    def __init__(self, num_particles):
        # Initialize particles with random poses
        self.particles = [Particle(x, y, theta) for _ in range(num_particles)]
        
    def motion_update(self, odom_delta):
        """Predict: Move particles based on odometry"""
        for p in self.particles:
            # Apply motion model with noise
            p.x += odom_delta.x + gaussian_noise(alpha3)
            p.y += odom_delta.y + gaussian_noise(alpha3)
            p.theta += odom_delta.theta + gaussian_noise(____)  # alpha1
    
    def sensor_update(self, laser_scan, map_):
        """Update: Weight particles by scan-map match"""
        for p in self.particles:
            # Project laser scan from particle pose
            expected_scan = ray_trace(p.pose, map_)
            
            # Compare to actual scan (likelihood field)
            p.weight = compute_likelihood(laser_scan, expected_scan)
        
        # Normalize weights
        total_weight = sum(p.weight for p in self.particles)
        for p in self.particles:
            p.weight /= ____  # total_weight
    
    def resample(self):
        """Resample particles proportional to weights"""
        new_particles = []
        for _ in range(len(self.particles)):
            # Low variance resampling
            selected = weighted_random_choice(self.particles)
            new_particles.append(copy(selected))
        
        self.particles = new_particles
    
    def estimate_pose(self):
        """Return weighted average of particles"""
        x = sum(p.x * p.weight for p in self.particles)
        y = sum(p.y * p.weight for p in self.particles)
        theta = circular_mean([p.theta for p in self.particles], 
                              [p.weight for p in self.particles])
        return Pose(x, y, theta)

Motion Model Types

Model Robot Type DOF
DifferentialMotionModel Differential drive (TurtleBot, etc.) 2D (x, y, θ)
OmniMotionModel Omnidirectional (mecanum wheels) 2D with strafe

Laser Model Types

Model Computation Accuracy
likelihood_field Fast (pre-computed distance field) Good for most cases
beam Slow (ray tracing per particle) Higher accuracy, handles dynamics

C++ Custom Initialization

#include <rclcpp/rclcpp.hpp>
#include <geometry_msgs/msg/pose_with_covariance_stamped.hpp>

class InitialPosePublisher : public rclcpp::Node {
public:
    InitialPosePublisher() : Node("initial_pose_pub") {
        publisher_ = this->create_publisher<
            geometry_msgs::msg::PoseWithCovarianceStamped>(
            "/initialpose", 10);
        
        // Set known starting pose
        auto msg = geometry_msgs::msg::PoseWithCovarianceStamped();
        msg.header.frame_id = "map";
        msg.header.stamp = this->now();
        
        // Position
        msg.pose.pose.position.x = 0.0;
        msg.pose.pose.position.y = 0.0;
        msg.pose.pose.orientation.w = 1.0;  // No rotation
        
        // Covariance (uncertainty)
        msg.pose.covariance[0] = 0.25;  // x variance
        msg.pose.covariance[7] = 0.25;  // y variance
        msg.pose.covariance[35] = 0.068;  // theta variance
        
        publisher_->publish(msg);
        RCLCPP_INFO(this->get_logger(), "Initial pose set");
    }

private:
    rclcpp::Publisher<geometry_msgs::msg::PoseWithCovarianceStamped>::SharedPtr publisher_;
};

Parameters & Config

Alpha Parameters (Motion Noise)

Tune based on odometry quality:

Parameter Meaning Increase When
alpha1 Rotation error from rotation Robot spins inaccurately
alpha2 Rotation error from translation Robot veers while driving straight
alpha3 Translation error from translation Wheel slip, rough terrain
alpha4 Translation error from rotation Robot drifts during turns

Particle Count Tuning

# Few particles (500-1000): Fast but less robust
min_particles: 500
max_particles: 1000
# Use for: Simple environments, good odometry, high CPU load

# Many particles (2000-5000): Slow but robust
min_particles: 2000
max_particles: 5000
# Use for: Symmetric environments, poor odometry, kidnapped robot

Update Frequency

# High update rate (small thresholds)
update_min_d: 0.1  # Update every 10cm
update_min_a: 0.2  # Update every ~11 degrees
# Pros: Accurate tracking, quick convergence
# Cons: High CPU usage

# Low update rate (large thresholds)
update_min_d: 0.5  # Update every 50cm
update_min_a: 1.0  # Update every ~57 degrees
# Pros: Low CPU usage
# Cons: Slower convergence, may lose track in fast motion

Monitor particle cloud convergence. On startup, particles spread across entire free space. As robot moves and receives scan updates, particles should converge to single cluster within 10-30 seconds.

Common Failures & Fix Patterns

Failure 1: Particles Diverge / No Convergence

Symptom: Particle cloud remains scattered, pose estimate jumps around, robot can't localize.

Root Cause: Poor odometry, insufficient distinctive features, or incorrect initial pose.

Fix: Increase particle count and motion noise:

# Allow more hypotheses
max_particles: 5000

# Increase motion noise to spread particles more
alpha1: 0.4  # Was 0.2
alpha2: 0.4
alpha3: 0.4
alpha4: 0.4

# Use more laser beams for better matching
max_beams: 180  # Was 60

# Set better initial pose (closer to truth)
# Use RViz 2D Pose Estimate tool interactively

Failure 2: Global Localization Failure

Symptom: Without good initial pose, AMCL never localizes (kidnapped robot problem).

Root Cause: Particle filter needs approximate starting pose, can't search entire map efficiently.

Fix: Use global localization service or increase initial spread:

# Request global localization (spreads particles everywhere)
ros2 service call /reinitialize_global_localization std_srvs/srv/Empty

# Or increase initial covariance in config
initial_pose:
  x: 0.0
  y: 0.0
  yaw: 0.0
  # Large covariance = wide particle spread
  covariance_x: ____  # 25.0 (spread over ±10m)
  covariance_y: 25.0
  covariance_yaw: 0.68  # ±60 degrees

# Drive robot around distinctive features to help convergence

Failure 3: TF Timeout / Extrapolation Errors

Symptom: "Transform timeout" or "extrapolation into future" errors, navigation fails.

Root Cause: AMCL not publishing map→odom transform fast enough or timing issues.

Fix: Increase transform tolerance and check timing:

# Increase tolerance for delayed transforms
transform_tolerance: 0.5  # Was 0.1 seconds

# Reduce update thresholds for more frequent publishing
update_min_d: 0.1
update_min_a: 0.1

# Check for timing issues
# Terminal 1:
ros2 topic hz /scan  # Should be stable (e.g., 10Hz)

# Terminal 2:
ros2 run tf2_ros tf2_monitor
# Look for high latency on map->odom or odom->base_link

# If odometry is delayed, fix odom publisher, not AMCL

Detection: Monitor pose covariance:

# pose_monitor.py
import rclpy
from rclpy.node import Node
from geometry_msgs.msg import PoseWithCovarianceStamped
import math

class PoseMonitor(Node):
    def __init__(self):
        super().__init__('pose_monitor')
        self.subscription = self.create_subscription(
            PoseWithCovarianceStamped,
            '/amcl_pose',
            self.pose_callback,
            10)
    
    def pose_callback(self, msg):
        # Extract position covariance
        cov_x = msg.pose.covariance[0]
        cov_y = msg.pose.covariance[7]
        
        uncertainty = math.sqrt(cov_x + cov_y)
        
        if uncertainty > ____:  # 1.0 meter
            self.get_logger().warn(f"High uncertainty: {uncertainty:.2f}m - possible kidnapping!")
        else:
            self.get_logger().info(f"Localized: {uncertainty:.2f}m")

Warning:

AMCL assumes static environment. Moving obstacles (people, other robots) can cause false localization if they occupy significant map area. Use obstacle filtering or dynamic map updates for highly dynamic environments.
