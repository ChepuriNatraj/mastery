Pointers & Memory Management

Prevent crashes, leaks, and undefined behavior in ROS2 C++ nodes by applying ownership rules and smart-pointer patterns correctly.

What / Why / When

What

C++ gives manual control over object lifetime. In robotics, this means you decide where data lives (stack/heap), who owns it, and when it is destroyed. ROS2 C++ APIs heavily use smart pointers to represent this ownership.

Why

Robot processes often run for hours. Small leaks accumulate into instability; dangling pointers cause random crashes during mission execution. Correct memory ownership is a reliability requirement, not just a style choice.

When

Use these rules in every callback, plugin, and node constructor. If ownership is shared across asynchronous callbacks, use shared_ptr. If ownership is exclusive, use unique_ptr. Use weak_ptr for back references and observer links.

10-Minute Drill

#include <memory>
#include <iostream>

int main() {
  int local_value = 10;  // stack

  auto unique_val = std::make_unique<int>(42);  // exclusive heap owner
  auto shared_a = std::make_shared<int>(7);     // shared owner
  auto shared_b = shared_a;                      // ref count +1
  std::weak_ptr<int> weak_view = shared_a;       // observer only

  std::cout << "shared count: " << shared_a.use_count() << "\n"; // ____ 2

  if (auto locked = weak_view.lock()) {
    std::cout << "weak lock value: " << *locked << "\n";
  }

  // move unique ownership
  auto unique_moved = std::move(unique_val);
  std::cout << std::boolalpha << (unique_val == nullptr) << "\n"; // true
}

Safe Callback Capture

class SafeNode : public rclcpp::Node, public std::enable_shared_from_this<SafeNode> {
public:
  SafeNode() : Node("safe_node") {
    timer_ = create_wall_timer(std::chrono::seconds(1), [this]() {
      RCLCPP_INFO(this->get_logger(), "tick");
    });
  }

  void start_guarded_timer() {
    auto self = shared_from_this();
    guarded_timer_ = create_wall_timer(std::chrono::seconds(1), [weak = std::weak_ptr<SafeNode>(self)]() {
      if (auto node = weak.lock()) {
        RCLCPP_INFO(node->get_logger(), "safe tick");
      }
    });
  }

private:
  rclcpp::TimerBase::SharedPtr timer_;
  rclcpp::TimerBase::SharedPtr guarded_timer_;
};

Cycle Leak Pattern

struct Parent;
struct Child {
  std::weak_ptr<Parent> parent; // weak avoids cycle
};

struct Parent {
  std::shared_ptr<Child> child;
};

auto p = std::make_shared<Parent>();
auto c = std::make_shared<Child>();
c->parent = ____; // p
p->child = c;

Common Failures & Fix Patterns

  • Use-after-free: callback captures raw this and outlives object. Fix using weak_ptr.lock().
  • Memory leak: shared_ptr cycles. Break with weak_ptr on one side.
  • Double delete: mixing raw owning pointers and smart pointers. Keep one clear owner.
  • Null dereference: moved unique_ptr still used. Check for null after std::move.

Verify

# Build with sanitizers (debug only)
colcon build --cmake-args -DCMAKE_BUILD_TYPE=Debug \
  -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined -fno-omit-frame-pointer" \
  -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=address,undefined"

# Run your node and watch for ASan output
ros2 run my_robot_cpp my_node

# Optional memory-leak check
valgrind --leak-check=full ros2 run my_robot_cpp my_node
