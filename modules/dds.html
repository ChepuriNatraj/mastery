DDS & Discovery

Master DDS middleware, discovery protocols, and multi-robot networking.

DDS Networking

What / Why / When

What

DDS (Data Distribution Service) is ROS2's middleware layer. It handles discovery (nodes finding each other), serialization (message encoding), and transport (UDP/TCP). ROS2 supports multiple DDS vendors: FastDDS (default), CycloneDDS, RTI Connext. Discovery uses multicast UDP to announce nodes/topics. Domain IDs isolate groups (domain 0 separate from domain 1). QoS policies control reliability, durability, history. Security plugins enable encryption, authentication. RMW (ROS Middleware) layer abstracts DDS vendor differences. Environment variables control DDS behavior: ROS_DOMAIN_ID, RMW_IMPLEMENTATION, FASTRTPS_DEFAULT_PROFILES_FILE.

Why

DDS provides peer-to-peer communication - no central master like ROS1. Nodes discover each other automatically using multicast. QoS enables fine-grained control - RELIABLE for commands, BEST_EFFORT for sensor streams. DDS scales to 1000+ nodes across multiple machines. Domain IDs prevent crosstalk between robots. DDS security prevents unauthorized access. Vendor choice matters: FastDDS has broad compatibility, CycloneDDS has lower latency, RTI Connext is enterprise-grade. Understanding DDS is critical for debugging network issues (nodes not discovering, messages lost).

When

Use separate domain IDs for each robot (0-101 for single robot, 102-232 for multi-robot). Change to CycloneDDS for lower latency (real-time control). Disable multicast discovery for large networks (use Simple Discovery or Static Discovery). Use DDS security for production robots (prevent hacking). Tune shared memory transport for inter-process communication (10x faster than UDP). Monitor DDS discovery with ros2 daemon stop + ros2 topic list. Profile DDS performance with built-in tracing (identify bottlenecks).

Discovery Protocol

# Discovery happens automatically via multicast
# 1. Node starts, sends SPDP (Simple Participant Discovery Protocol)
#    "I am node X on domain 0"
# 2. Other nodes receive, send their own SPDP
# 3. Nodes exchange SEDP (Simple Endpoint Discovery Protocol)
#    "I publish topic /cmd_vel with type geometry_msgs/Twist"
# 4. Matching: If pub/sub QoS compatible, connection established

# Multicast addresses used:
# SPDP: 239.255.0.1:7400 + domain_id
# SEDP: Unicast back to discovered nodes

# Discovery takes 1-3 seconds
# Use: ros2 topic list  (waits for discovery)

Domain ID Configuration

# Set domain ID (0-101 for single domain, 102-232 for multi-domain)
export ROS_DOMAIN_ID=____  # 42

# Verify current domain
echo $ROS_DOMAIN_ID

# Run node in specific domain
ROS_DOMAIN_ID=5 ros2 run my_pkg my_node

# Only nodes in same domain see each other
# Domain 0: Robot A
# Domain 1: Robot B (isolated from A)

Switch DDS Implementation

# Check current DDS vendor
ros2 doctor --report | grep middleware

# Install CycloneDDS
sudo apt install ros-humble-rmw-cyclonedds-cpp

# Use CycloneDDS
export RMW_IMPLEMENTATION=____  # rmw_cyclonedds_cpp
ros2 run my_pkg my_node

# Use FastDDS (default)
export RMW_IMPLEMENTATION=rmw_fastrtps_cpp

# Make permanent in ~/.bashrc
echo 'export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp' >> ~/.bashrc

Concept ↔ Code (DDS Configuration)

FastDDS XML Profile

<!-- fastdds_profile.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<dds>
  <profiles xmlns="http://www.eprosima.com/XMLSchemas/fastRTPS_Profiles">
    
    <!-- Default participant profile -->
    <participant profile_name="default_participant" is_default_profile="true">
      
      <!-- Domain ID -->
      <domainId>0</domainId>
      
      <!-- Discovery config -->
      <rtps>
        <builtin>
          <discovery_config>
            
            <!-- Simple discovery (default) -->
            <discoveryProtocol>SIMPLE</discoveryProtocol>
            
            <!-- Increase discovery retries for WiFi -->
            <leaseDuration>
              <sec>____</sec>  <!-- 10 seconds -->
            </leaseDuration>
            
            <!-- Multicast locators -->
            <metatrafficMulticastLocatorList>
              <locator>
                <udpv4>
                  <address>239.255.0.1</address>
                  <port>7400</port>
                </udpv4>
              </locator>
            </metatrafficMulticastLocatorList>
            
          </discovery_config>
        </builtin>
        
        <!-- Use shared memory for local communication -->
        <useBuiltinTransports>false</useBuiltinTransports>
        <userTransports>
          <transport_id>shm_transport</transport_id>
          <transport_id>udp_transport</transport_id>
        </userTransports>
        
      </rtps>
    </participant>
    
    <!-- Shared memory transport (10x faster for local nodes) -->
    <transport_descriptors>
      <transport_descriptor>
        <transport_id>shm_transport</transport_id>
        <type>SHM</type>
      </transport_descriptor>
      
      <transport_descriptor>
        <transport_id>udp_transport</transport_id>
        <type>UDPv4</type>
      </transport_descriptor>
    </transport_descriptors>
    
  </profiles>
</dds>
# Use profile
export FASTRTPS_DEFAULT_PROFILES_FILE=/path/to/fastdds_profile.xml
ros2 run my_pkg my_node

CycloneDDS Configuration

<!-- cyclonedds.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<CycloneDDS>
  <Domain>
    <Id>0</Id>
    
    <Discovery>
      <!-- Multicast discovery -->
      <EnableMulticast>true</EnableMulticast>
      
      <!-- Discovery peers (for static discovery) -->
      <Peers>
        <Peer address="192.168.1.10"/>
        <Peer address="____"/>  <!-- 192.168.1.11 (add peers manually) -->
      </Peers>
    </Discovery>
    
    <Tracing>
      <!-- Enable DDS tracing for debugging -->
      <Verbosity>info</Verbosity>  <!-- none, severe, error, warning, info, config, fine, finer, finest -->
      <OutputFile>cyclonedds.log</OutputFile>
    </Tracing>
    
  </Domain>
</CycloneDDS>
# Use CycloneDDS config
export CYCLONEDDS_URI=file:///path/to/cyclonedds.xml
export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp
ros2 run my_pkg my_node

Static Discovery (No Multicast)

For networks where multicast is disabled/unreliable:

<!-- fastdds_static.xml -->
<discoveryProtocol>____</discoveryProtocol>  <!-- SIMPLE or SERVER (new in Humble) -->

<!-- Discovery Server mode (scalable for 100+ nodes) -->
<discoveryServersList>
  <RemoteServer prefix="44.53.00.5f.45.50.52.4f.53.49.4d.41">
    <metatrafficUnicastLocatorList>
      <locator>
        <udpv4>
          <address>192.168.1.100</address>  <!-- Server IP -->
          <port>11811</port>
        </udpv4>
      </locator>
    </metatrafficUnicastLocatorList>
  </RemoteServer>
</discoveryServersList>
# Launch discovery server
fastdds discovery -i 0 -l 192.168.1.100 -p 11811

# Clients connect to server (no multicast needed)
export ROS_DISCOVERY_SERVER="192.168.1.100:11811"
ros2 run my_pkg my_node

DDS Security (Encryption & Auth)

# Generate security files
ros2 security create_keystore demo_keys
ros2 security create_enclave demo_keys /my_robot/my_node

# Creates:
# demo_keys/enclaves/my_robot/my_node/
#   identity_ca.cert.pem       # CA certificate
#   cert.pem                   # Node certificate
#   key.pem                    # Private key
#   permissions_ca.cert.pem    # Permissions CA
#   governance.p7s             # Governance rules
#   permissions.p7s            # Node permissions

# Enable security
export ROS_SECURITY_KEYSTORE=/path/to/demo_keys
export ROS_SECURITY_ENABLE=true
export ROS_SECURITY_STRATEGY=____  # Enforce (or Permissive for testing)

ros2 run my_pkg my_node

# Machine B (same network)

ros2 run demo_nodes_cpp listener

# If not:
# 1. Check firewall (allow UDP 7400-7500)
# 2. Check multicast routing (ping 239.255.0.1)

Domain Isolation Test

# Terminal 1
export ROS_DOMAIN_ID=0
ros2 topic pub /test std_msgs/String "data: 'Domain 0'"

# Terminal 2 (same domain)
export ROS_DOMAIN_ID=0
ros2 topic echo /test
# Should receive "Domain 0"

# Terminal 3 (different domain)
export ROS_DOMAIN_ID=1
ros2 topic echo /test
# Should NOT receive (isolated)

Use ros2 doctor --report for comprehensive DDS health check. It reports middleware, domain ID, network interfaces, and common issues.

Common Failures & Fix Patterns

Failure 1: Nodes Don't Discover Each Other

Symptom: ros2 node list doesn't show remote nodes.

Root Cause: Firewall blocking multicast, wrong domain ID, or network isolation.

Fix: Check connectivity and firewall:

# Check domain ID matches
echo $ROS_DOMAIN_ID  # Both machines must match

# Test multicast connectivity
# Machine A
ping -I eth0 239.255.0.1

# Machine B (should receive pings)
# If not, multicast routing broken

# Check firewall (Ubuntu)
sudo ufw allow 7400:7500/udp  # Allow DDS discovery

# Or disable temporarily for testing
sudo ufw disable

# Verify discovery
ros2 daemon stop
ros2 node list

Failure 2: High Latency / Packet Loss

Symptom: Messages delayed or dropped, ros2 topic hz shows irregular rate.

Root Cause: WiFi congestion, QoS mismatch, or insufficient bandwidth.

Fix: Tune QoS and transport:

# Switch to CycloneDDS (lower latency)
export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp

# For local nodes, use shared memory (10x faster)
# Add to FastDDS XML:
<useBuiltinTransports>false</useBuiltinTransports>
<userTransports>
  <transport_id>____</transport_id>  # shm_transport
</userTransports>

# Monitor topic rate
ros2 topic hz /scan
# Should be stable (e.g., 10.0 Hz ± 0.1)

Failure 3: "Failed to Create Node" Error

Symptom: Node crashes with "failed to create node" or "failed to initialize middleware".

Root Cause: DDS implementation mismatch or missing shared memory.

Fix: Reset DDS state:

# Clear shared memory segments
rm -rf /dev/shm/fastrtps_*
rm -rf /dev/shm/sem.fastrtps_*

# Check RMW matches installed packages
ros2 doctor --report | grep rmw
# If rmw_cyclonedds_cpp but package not installed:
sudo apt install ros-humble-____  # rmw-cyclonedds-cpp

# Restart daemon
ros2 daemon stop
ros2 daemon start

Failure Injection Drill

Exercise: Domain ID Mismatch

Create publisher and subscriber in different domains:

# Terminal 1 - Publisher (domain 0)
export ROS_DOMAIN_ID=____  # 0
ros2 run demo_nodes_cpp talker

# Terminal 2 - Subscriber (domain 5) - intentional error
export ROS_DOMAIN_ID=5
ros2 run demo_nodes_cpp listener

Observe:

  • Publisher sends "Hello World: X"
  • Subscriber receives nothing (different domain)
  • ros2 node list in domain 5 doesn't show talker
  • No error messages - silent failure!

Detection:

# Check domains
# Terminal 1
echo $ROS_DOMAIN_ID  # 0

# Terminal 2
echo $ROS_DOMAIN_ID  # 5 (mismatch!)

# Verify isolation
ROS_DOMAIN_ID=0 ros2 topic list | grep chatter  # Found
ROS_DOMAIN_ID=5 ros2 topic list | grep chatter  # Not found

Recovery:

# Fix domain ID
export ROS_DOMAIN_ID=____  # 0 (match publisher)
ros2 run demo_nodes_cpp listener
# Now receives messages!
Warning:

Domain ID mismatch is silent - no error, just no communication. Always verify ROS_DOMAIN_ID matches across all terminals/machines. Add to ~/.bashrc to make consistent.
