Actions

Long-running tasks with feedback and cancellation support.

CPU-Light Required

What / Why / When

What

Actions combine pub/sub and services for preemptable, long-running tasks. They provide goal submission, continuous feedback, and result delivery. Clients can cancel goals mid-execution. Actions use three topics internally: goal, feedback, and result. Nav2's navigate-to-pose is the canonical action example.

Why

Services block until completion, unsuitable for tasks taking seconds or minutes. Topics provide no acknowledgment or completion status. Actions solve this by allowing non-blocking goal submission with progress updates and guaranteed result delivery. They're essential for navigation, manipulation, and any interruptible task.

When

Use actions for any operation over 1 second where clients need progress feedback or cancellation ability. Navigation goals, arm movements, and autonomous tasks fit perfectly. Avoid actions for instant operations (use services) or continuous streams (use topics). Actions add overhead, so reserve them for truly long-running work.

C++ Micro-Task

Create a Fibonacci action server:

#include "rclcpp/rclcpp.hpp"
#include "rclcpp_action/rclcpp_action.hpp"
#include "example_interfaces/action/fibonacci.hpp"

using Fibonacci = example_interfaces::action::Fibonacci;
using GoalHandle = rclcpp_action::ServerGoalHandle<Fibonacci>;

class FibonacciServer : public rclcpp::Node {
public:
  FibonacciServer() : Node("fibonacci_server") {
    action_server_ = rclcpp_action::create_server<Fibonacci>(
      this, "____",
      std::bind(&FibonacciServer::handle_goal, this, _1, _2),
      std::bind(&FibonacciServer::handle_cancel, this, _1),
      std::bind(&FibonacciServer::handle_accepted, this, _1));
  }

private:
  rclcpp_action::GoalResponse handle_goal(
    const rclcpp_action::GoalUUID& uuid,
    std::shared_ptr<const Fibonacci::Goal> goal) {
    RCLCPP_INFO(this->get_logger(), "Received goal order %d", goal->order);
    return rclcpp_action::GoalResponse::____; // ACCEPT_AND_EXECUTE
  }

  rclcpp_action::CancelResponse handle_cancel(
    const std::shared_ptr<GoalHandle> goal_handle) {
    RCLCPP_INFO(this->get_logger(), "Cancel request");
    return rclcpp_action::CancelResponse::____; // ACCEPT
  }

  void handle_accepted(const std::shared_ptr<GoalHandle> goal_handle) {
    std::thread{std::bind(&FibonacciServer::execute, this, _1), goal_handle}.detach();
  }

  void execute(const std::shared_ptr<GoalHandle> goal_handle) {
    auto result = std::make_shared<Fibonacci::Result>();
    auto feedback = std::make_shared<Fibonacci::Feedback>();
    auto& sequence = feedback->sequence;
    sequence.push_back(0);
    sequence.push_back(1);

    for (int i = 1; i < goal_handle->get_goal()->order; ++i) {
      if (goal_handle->is_canceling()) {
        goal_handle->canceled(result);
        return;
      }
      sequence.push_back(sequence[i] + sequence[i - 1]);
      goal_handle->publish_feedback(____);
      std::this_thread::sleep_for(std::chrono::milliseconds(500));
    }
    result->sequence = sequence;
    goal_handle->succeed(result);
  }

  rclcpp_action::Server<Fibonacci>::SharedPtr action_server_;
};

int main(int argc, char **argv) {
  rclcpp::init(argc, argv);
  rclcpp::spin(std::make_shared<FibonacciServer>());
  rclcpp::shutdown();
  return 0;
}

from example_interfaces.action import Fibonacci

    def send_goal(self, order):

        self.action_client.wait_for_server()
        self.send_goal_future = self.action_client.send_goal_async(
            goal_msg, feedback_callback=self.feedback_callback)
        self.send_goal_future.add_done_callback(self.goal_response_callback)

    def feedback_callback(self, feedback_msg):
        feedback = feedback_msg.feedback
        self.get_logger().info('Feedback: %s' % feedback.____)

    def goal_response_callback(self, future):
        goal_handle = future.result()
        if not goal_handle.accepted:
            self.get_logger().info('Goal rejected')
            return
        self.get_logger().info('Goal accepted')
        self.result_future = goal_handle.get_result_async()
        self.result_future.add_done_callback(self.result_callback)

    def result_callback(self, future):
        result = future.result().result
        self.get_logger().info('Result: %s' % result.sequence)

def main(args=None):
    rclpy.init(args=args)
    client = FibonacciClient()
    client.send_goal(10)
    rclpy.spin(client)
    rclpy.shutdown()

if __name__ == '__main__':
    main()

Common Failures & Fix Patterns

Failure 1: Goal Rejected Immediately

Symptom: Server rejects goal without executing, client receives rejection.

Root Cause: Server's handle_goal returns REJECT due to validation failure or busy state.

Fix: Check goal validation logic and server capacity:

rclcpp_action::GoalResponse handle_goal(...) {
  if (goal->order < 1 || goal->order > 100) {
    RCLCPP_WARN(this->get_logger(), "Invalid order: %d", goal->order);
    return rclcpp_action::GoalResponse::REJECT;
  }
  if (current_goals_ >= max_goals_) {
    return rclcpp_action::GoalResponse::REJECT;
  }
  return rclcpp_action::GoalResponse::ACCEPT_AND_EXECUTE;
}

Failure 2: Feedback Never Received

Symptom: Goal executes but client's feedback callback never fires.

Root Cause: Client didn't register feedback callback or server doesn't publish feedback.

Fix: Ensure both sides configured correctly:

# Client must specify feedback callback
future = self.action_client.send_goal_async(
    goal_msg,
    feedback_callback=self.feedback_callback)  # Don't forget this!

# Server must publish feedback
goal_handle.publish_feedback(feedback_msg)  # Inside execute loop

Failure 3: Cancel Request Ignored

Symptom: Client cancels goal but server continues executing.

Root Cause: Server's execute loop doesn't check is_canceling() status.

Fix: Poll cancellation state in tight loops:

void execute(const std::shared_ptr<GoalHandle> goal_handle) {
  while (work_remaining) {
    if (goal_handle->is_canceling()) {
      RCLCPP_INFO(this->get_logger(), "Goal canceled");
      goal_handle->canceled(result);
      return;  // Exit immediately
    }
    // Do work...
  }
  goal_handle->succeed(result);
}

Failure Injection Drill

Exercise: Timeout and Retry

Create a server that randomly fails goals:

import random

def execute_callback(self, goal_handle):
    if random.random() < 0.3:  # 30% failure rate
        self.get_logger().error('Simulated failure')
        goal_handle.abort()
        return ActionType.Result()
    
    # Normal execution...
    goal_handle.succeed()
    return result

Detection: Client checks result status:

def result_callback(self, future):
    status = future.result().status
    if status == GoalStatus.STATUS_ABORTED:
        self.get_logger().warn('Goal aborted, retrying...')
        self.retry_count += 1
        if self.retry_count < 3:
            self.send_goal(self.last_goal)
    elif status == GoalStatus.STATUS_SUCCEEDED:
        self.get_logger().info('Success!')
        result = future.result().result

Recovery: Implement exponential backoff for retries.

Warning:

Always set timeouts on action clients. Network issues or server crashes can leave clients waiting indefinitely.
