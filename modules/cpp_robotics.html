C++ for Robotics

Master the C++ patterns that appear everywhere in ROS2 and MoveIt2 codebases.

C++17 Required CMake

What / Why / When

What

ROS2 and MoveIt2 are written almost entirely in modern C++ (C++17). Key patterns you encounter daily: smart pointers (shared_ptr, unique_ptr), std::bind and lambdas for callbacks, RAII for resource management, templates for generic message types, threading with std::thread and mutexes, and std::chrono for time. ROS2 adds its own idioms: SharedPtr type aliases on every node class, create_wall_timer with std::bind, and executor patterns for spin control. Understanding these patterns is mandatory for reading and writing production ROS2 C++ code.

Why

Python ROS2 nodes are simpler to write but hit CPU limits fast: laser scan processing at 40Hz with Python is ~10x slower than C++. Real-time control loops, point cloud processing, and motion planning all require C++. C++ also provides: zero-cost abstractions (templates, inline), deterministic memory layout (structs), RAII to prevent resource leaks in long-running nodes, and compile-time error detection. Most nav2 and MoveIt2 plugins are C++-only. Understanding C++ patterns lets you read the Nav2 source, write custom planners, and optimize bottlenecks.

When

Use C++ for ROS2 when: processing high-frequency sensor data (lidar, cameras), writing custom Nav2/MoveIt2 plugins, implementing real-time control loops, or maximizing computational throughput. Use Python for: rapid prototyping, launch files, simple parameter servers, and glue logic. A common pattern is prototyping in Python, then rewriting hot paths in C++. Don't over-optimize — profile first. Use C++ only where Python is a proven bottleneck (use ros2 topic hz and top to identify CPU-bound nodes).

10-Minute Drill

C++ Micro-Task: Smart Pointers in Nodes

Practice the shared_ptr patterns that appear in every ROS2 node:

#include "rclcpp/rclcpp.hpp"
#include "std_msgs/msg/float64.hpp"

class SensorNode : public rclcpp::Node {
public:
  // static factory: always create nodes via make_shared
  SensorNode() : Node("sensor_node") {
    // shared_ptr from this — use shared_from_this() instead of 'this'
    publisher_ = this->create_publisher<std_msgs::msg::Float64>("sensor", 10);

    // std::bind creates callback: binds member function + 'this' pointer
    timer_ = this->create_wall_timer(
      std::chrono::milliseconds(____),  // 100
      std::bind(&SensorNode::publish_reading, this));
  }

private:
  void publish_reading() {
    auto msg = std::make_unique<std_msgs::msg::Float64>();
    msg->data = get_reading();
    publisher_->publish(std::move(____));  // msg
  }

  double get_reading() { return 23.5; }

  // Member variables are SharedPtr — owned for node lifetime
  rclcpp::Publisher<std_msgs::msg::Float64>::SharedPtr publisher_;
  rclcpp::TimerBase::SharedPtr timer_;
};

int main(int argc, char* argv[]) {
  rclcpp::init(argc, argv);
  // make_shared creates SensorNode AND its control block in one allocation
  rclcpp::spin(std::make_shared<____>());  // SensorNode
  rclcpp::shutdown();
}

Core C++ Patterns for ROS2

RAII — Resource Acquisition Is Initialization

Pattern ROS2 Usage
unique_ptr Exclusive ownership: plugin instances, hardware handles
shared_ptr Shared ownership: node handles, publishers, subscribers
lock_guard Auto-unlock mutex: protecting shared state in callbacks
Destructor cleanup Stop hardware interfaces when node is destroyed 
// Thread-safe callback with RAII lock_guard
class SafeNode : public rclcpp::Node {
  void callback(const std_msgs::msg::Float64::SharedPtr msg) {
    std::lock_guard<std::mutex> lock(mutex_);  // auto-unlocks on scope exit
    shared_data_ = msg->data;
  }
  std::mutex mutex_;
  double shared_data_ = 0.0;
};

Templates in ROS2

// Generic publisher factory — reuse across message types
template<typename MsgT>
rclcpp::Publisher<MsgT>::SharedPtr make_latched_pub(
    rclcpp::Node* node, const std::string& topic) {
  auto qos = rclcpp::QoS(1).transient_local();
  return node->create_publisher<MsgT>(topic, qos);
}

// Usage:
auto pose_pub = make_latched_pub<geometry_msgs::msg::PoseStamped>(this, "/goal");
auto scan_pub = make_latched_pub<sensor_msgs::msg::LaserScan>(this, "/scan");

Parameters & Config

Declaring and Using Parameters in C++

class ParamNode : public rclcpp::Node {
public:
  ParamNode() : Node("param_node") {
    // Declare with default + descriptor
    rcl_interfaces::msg::ParameterDescriptor desc;
    desc.description = "Maximum velocity in m/s";
    desc.floating_point_range.resize(1);
    desc.floating_point_range[0].from_value = 0.0;
    desc.floating_point_range[0].to_value = 2.0;

    this->declare_parameter("max_vel", 0.5, desc);
    this->declare_parameter("robot_name", std::string("robot1"));
    this->declare_parameter("sensor_ids", std::vector<int64_t>{1, 2, 3});

    // Read parameters
    max_vel_ = this->get_parameter("____").as_double();  // "max_vel"
    robot_name_ = this->get_parameter("robot_name").as_string();

    // Register dynamic reconfigure callback
    param_cb_ = this->add_on_set_parameters_callback(
      [this](const std::vector<rclcpp::Parameter>& params) {
        rcl_interfaces::msg::SetParametersResult result;
        result.successful = true;
        for (const auto& p : params) {
          if (p.get_name() == "max_vel") {
            if (p.as_double() < 0.0) {
              result.successful = false;
              result.reason = "max_vel must be non-negative";
            } else {
              max_vel_ = p.as_double();
            }
          }
        }
        return result;
      });
  }

private:
  double max_vel_;
  std::string robot_name_;
  rclcpp::node_interfaces::OnSetParametersCallbackHandle::SharedPtr param_cb_;
};

Common Failures & Fix Patterns

Failure 1: Use-After-Free with Raw this Pointer in Callbacks

Symptom: Segfault or corrupted data when node is destroyed while callback is running.

Root Cause: Lambda captures raw this pointer; node destroyed while timer fires.

Fix: Use shared_from_this() to keep node alive during callback execution:

// WRONG: raw 'this' can dangle
timer_ = create_wall_timer(100ms, [this]() { process(); });

// CORRECT: weak_ptr prevents use-after-free
auto weak = weak_from_this();
timer_ = create_wall_timer(std::chrono::milliseconds(100), [weak]() {
  if (auto node = weak.lock()) {
    node->process();
  }
});

Failure 2: Data Race in Multi-Executor Setup

Symptom: Random crashes or inconsistent data when using MultiThreadedExecutor.

Root Cause: Multiple callbacks accessing shared member variables without synchronization.

Fix: Protect shared state with mutex or use callback groups:

// Option A: mutex for shared state
std::mutex data_mutex_;
double shared_value_ = 0.0;

void subscriber_callback(const std_msgs::msg::Float64::SharedPtr msg) {
  std::lock_guard<std::mutex> lock(data_mutex_);
  shared_value_ = msg->data;
}

// Option B: MutuallyExclusive callback group (ensures serial execution)
auto cb_group = create_callback_group(
  rclcpp::CallbackGroupType::MutuallyExclusive);
rclcpp::SubscriptionOptions opts;
opts.callback_group = cb_group;
sub_ = create_subscription<std_msgs::msg::Float64>("data", 10, callback, opts);

Failure 3: Unresolved Linker Errors with ament_target_dependencies

Symptom: undefined reference to rclcpp::Node::create_publisher at link time.

Root Cause: Missing dependency in ament_target_dependencies() or wrong target name.

Fix: Verify CMakeLists.txt lists all packages used in source files:

# List EVERY package whose headers you #include
ament_target_dependencies(sensor_node
  rclcpp          # rclcpp/rclcpp.hpp
  std_msgs        # std_msgs/msg/float64.hpp
  geometry_msgs   # geometry_msgs/msg/pose.hpp
  sensor_msgs     # sensor_msgs/msg/laser_scan.hpp
)
# Missing a package here = linker error

Failure Injection Drill

Exercise: Trigger a Shared Pointer Cycle

Intentionally create a shared_ptr cycle and observe the memory leak:

// Step 1: Create circular reference (memory leak)
struct NodeA;
struct NodeB;

struct NodeA {
  std::shared_ptr<NodeB> b_ref;
  ~NodeA() { std::cout << "NodeA destroyed\n"; }
};

struct NodeB {
  std::shared_ptr<NodeA> a_ref;  // CYCLE: A owns B, B owns A
  ~NodeB() { std::cout << "NodeB destroyed\n"; }
};

{
  auto a = std::make_shared<NodeA>();
  auto b = std::make_shared<NodeB>();
  a->b_ref = b;
  b->a_ref = ____;  // a  -- creates cycle
}
// Observe: destructors NOT called (memory leaked)

// Step 2: Fix with weak_ptr
struct NodeB_Fixed {
  std::weak_ptr<NodeA> a_ref;  // weak_ptr breaks cycle
  ~NodeB_Fixed() { std::cout << "NodeB_Fixed destroyed\n"; }
};
// Now destructors ARE called in correct order

Detection: Use Valgrind to confirm the leak:

valgrind --leak-check=full ./your_program
# With cycle: "definitely lost: 2 blocks"
# With weak_ptr fix: "All heap blocks were freed"
Warning:

In ROS2 nodes, circular shared_ptr references between nodes and their callbacks are a common source of memory leaks. Always use weak_from_this() or std::weak_ptr when storing back-references to the owning node.
